I2C2 "双主机抢总线" 是什么意思

STM32F429ZG 电机调试板 · 软件 bit-bang I2C2 · 2026-06-04 · 配合扫描时 CLK 异常波形的诊断

一句话：同一颗 MCU 里有两段代码（主循环的扫描、TIM6 中断里的采样）都把自己当成 I2C2 的唯一主机，去操作同一对引脚 PB10/PB11，互相踩踏，把总线时序搅乱。不是两块芯片，也不是标准多主机仲裁。

1. 先看 I2C 总线的前提

I2C 只有两根线，都是 开漏输出 + 外部上拉电阻：

信号谁驱动作用
SCL 时钟正常只有主机驱动决定何时采样一个 bit、控制整笔事务的节奏
SDA 数据主机 / 从机分时驱动承载地址、数据、ACK

正常一条总线上只允许一个主机。它独占地决定一笔事务的完整边界：START → 地址 → 数据 → STOP。从机只能被动应答，绝不会主动发起时钟。

2. 本项目的"双主机"其实是两段并发代码

关键：不是两块芯片。是 STM32 内部两段独立执行流，都对同一对引脚做软件 bit-bang：

代码运行上下文优先级
主机 ABSP_I2C2_Scan → ProbeAddr主循环（扫描指令 0x07）—
主机 BApp_Motor_ReadReg（app_sample.c:275）TIM6 采样中断7

因为 I2C2 是软件 bit-bang——用 GPIO 手动把 SCL/SDA 拉高拉低，而 TIM6 中断可以在任意时刻打断主循环。两段代码都在"裸写"同一对引脚，谁都不知道对方正处于事务中途，也没有任何互斥。

3. 物理上发生了什么（碰撞时间线）

主循环（扫描 A） …发到第 3 个 bit，刚把 SCL 拉低… │ ▼ ← TIM6 中断在这一刻抢占 TIM6 中断（采样 B） ├─ 自己跑一整笔事务： │ START → 地址 → 寄存器地址 → 重复START → 读数据 → STOP │ （在同一 SCL/SDA 上来回翻转十几次） ▼ ← 中断返回 主循环（扫描 A） …以为还在第 3 个 bit，继续翻转 SCL…

从机和逻辑分析仪看到的，就是两笔事务的时钟边沿交织在一起：

  • 时钟"很奇怪"：本该是「9 个脉冲一簇（地址字节+ACK）+ 规整间隔」的扫描，却冒出孤立窄脉冲和长间隔 —— 那是采样事务的时钟插进了扫描事务里。
  • 解码乱码：一次 ProbeAddr 探测只发 1 个地址字节，绝不会有数据字节。但解码出现 addr 00 + Wr 后跟 Data write:00 / DW:83 / 00 / 03 多个数据字节 —— 那些就是采样事务的地址/数据字节，因为中间没有干净的 STOP 分隔，被解码器并进了同一帧。
  • 地址被解成 0x00：两边同时把 SDA 拉低，8 个 bit 全成 0。
打个比方：两个人同时用同一支笔在同一张纸上写字，或者两个人同时按下同一个对讲机讲话 —— 内容必然糊成一团。

为什么"只有一块板子"出现？

只有那块板子在你点扫描的瞬间恰好正在采样 / 跑波形发生器（在 I2C2 上持续流数据），所以主机 B 是活跃的。其它板子空闲，主机 B 没在跑，两者不冲突，因此波形正常。

4. 和"标准 I2C 多主机仲裁"的区别

标准多主机（两块芯片）本项目（两段并发代码）
是否有仲裁有：开始前检测总线空闲，发送时逐 bit 比对 SDA，谁让步谁退出无：纯软件 bit-bang，两段代码之间没有互斥/检测，直接互相踩踏
结果失败方干净退出，赢家继续边沿交织，两笔事务都损坏
解决思路靠硬件仲裁靠软件保证"两者不同时上总线"

5. 解决办法：让两者不要同时上总线

事务类型命令策略原因
单次短事务0x20/0x21/0x30/0x31AcquireBus 临时屏蔽 TIM6 中断（让主机 B 闭嘴几十 µs）事务短，屏蔽 ISR 时间可接受
长事务0x22 bulk · 0x07 扫描（本次漏掉）入口直接拒绝："采样/发生器在跑就别做"126 次探测耗时 ~25ms+，屏蔽 ISR 太久会丢采样点
本次根因：HandleI2CScan（0x07）对 I2C2 扫描既没有 AcquireBus，也没有像 0x22（app_protocol.c:561）那样在入口拒绝采样/发生器。补上同样的入口拒绝即可闭环。
