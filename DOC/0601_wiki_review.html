Flash 文件存储写 session(0x39→0x3B)期间未阻止采样/发生器启动 → I2C2 与片内 Flash 编程并发
📦 App-Protocol (app_protocol.c / app_protocol.h)  ·  📍 app_protocol.c:1361 (DispatchFrame) / 1168 HandleFlashStoreWriteBegin / 1201 HandleFlashStoreWriteData  ·  🗳️ 验证 2/2 支持
问题：s_flashSessionActive 只由 AW ISP 0x32 BEGIN 置位(1036)，0x39 flashstore BEGIN 不置任何 session 闸门，仅在 BEGIN/WIPE 入口一次性检查 sample/generator 未运行。0x39 之后、0x3B END 之前，DispatchFrame 仍放行 0x50 START_SAMPLE 与 0x55/0x56/0x58 生成器启动。一旦采样启动，TIM6 ISR 驱动 I2C2 软件 bit-bang，而 0x3A WriteData 调 App_FlashStore_WriteData 编程片内 Flash——STM32F4 Flash 编程/擦除会阻塞 Flash 总线，从 Flash 取指的 TIM6 ISR 在此期间无法执行,破坏采样时序甚至取指异常。属真实并发缺口,非已记录的 H-e。
建议：为 flashstore 写 session 增加独立 active 标志(如 s_storeSessionActive)，在 0x39 成功后置位、0x3B/错误清零，并在 DispatchFrame 顶部像 AW session 一样拒绝该期间的 sample/generator/0x30/0x31/bulk 命令;或在 0x50/0x55/0x56/0x58 入口追加 flashstore-session 拒绝判断。
RX 状态机无半截帧超时,错误 LEN 会吞掉后续整帧
📦 App-Protocol (app_protocol.c / app_protocol.h)  ·  📍 app_protocol.c:1515 ProcessByte (STATE_WAIT_DATA 1561)  ·  🗳️ 验证 2/2 支持
问题：若线路噪声/丢字节导致进入 STATE_WAIT_DATA 后期望的 len 字节迟迟收不齐，状态机无限期停留，期间到来的真正 0xAA 0x55 帧头被当作 DATA 吞掉，直到凑满 len 字节才进入 CRC 比对(必失败发 0x01)。一个被破坏的 LEN(如 0xFF)会连吃 255+2 字节，吞掉多个后续合法帧。PC 端 100ms 超时重发可缓解但会放大单次误码的影响面。
建议：引入帧内字节间超时(用 BSP_GetTick 记录上次收字节时刻,Poll 中若状态≠WAIT_SOF1 且超时则复位 s_state=STATE_WAIT_SOF1);或保留 SOF 双字节再同步作为兜底已部分覆盖,但 DATA 态无法借此恢复。
H-e：双缓冲数据与元数据可能不同源（通道错位/数据撕裂）
📦 App-Sample / App-Motor（示波器采样 + 波形发生器 + 电机 IC 寄存器访问）  ·  📍 App/Src/app_sample.c:432-447  ·  🗳️ 验证 2/2 支持
问题：Poll 在临界区(__disable_irq/__enable_irq, :433-439)只快照了 mask/failCnt/totalCnt/readIdx 并清 ready，但真正读取数据 buffer 的 SendStreamFrame(s_isrBuf[readIdx],...) 在 :447 临界区之外执行。SendStreamFrame 是纯内存拷贝+UART_Transmit 排队，本身较快，但若两次 TIM6 ISR 在此期间抢占（采样频率高/Poll 被其它 Poll 拖慢时可能），ISR 会再次写回 readIdx 这块 buffer（writeIdx 在两次翻转后回到 readIdx），导致 SendStreamFrame 读到的数据是新一帧的、而 mask/totalCount 仍是旧快照——极端时序下通道数据与掩码错位。这是 CLAUDE.md TODO 已登记的 H-e。
建议：两种修复其一：①在临界区内把 s_isrBuf[readIdx] 的有效字节一次性 memcpy 到本地/发送缓冲再出临界区组帧；②采用三缓冲或在 ISR 侧检测 readIdx 即将被覆盖时跳过(dataReady 仍为 1 表示主循环未取走则不翻转覆盖)。推荐方案①，拷贝量最大 16 字节，临界区开销可控。
发生器写返回值全部 (void) 丢弃，I2C 写失败上位机无感知
📦 App-Sample / App-Motor（示波器采样 + 波形发生器 + 电机 IC 寄存器访问）  ·  📍 App/Src/app_sample.c:117,151,200  ·  🗳️ 验证 2/2 支持
问题：Generator_DoLinearWrite/Sawtooth/Cosine 对 App_Motor_WriteReg 返回值一律 (void) 丢弃。发生器在 ISR 内运行不便直接回包，但若电机 IC 持续 NACK，上位机完全无法察觉写失败(不像采样有 fail 计数+auto-stop)。
建议：参照采样的失败计数机制，为发生器加一个连续写失败计数，超阈值发 SendDebugInfo 告警(类似 :354)，避免静默失败。
文件写会话(WriteData/WriteEnd)无协议层并发互斥，sampling/generator 可在写入中途启动
📦 App-FlashStore（片上 Flash 用户文件存储）  ·  📍 app_protocol.c:1200-1249 / app_flashstore.c:147-217  ·  🗳️ 验证 2/2 支持
问题：0x39 WRITE_BEGIN 和 0x3F WIPE 在入口拒绝 sampling/generator/bulkRead 并发(因 3-7s 擦区阻塞)，但 0x3A WRITE_DATA / 0x3B WRITE_END 没有任何互斥检查。一个文件写会话(s_writeActive=1)可能持续数秒到数十秒(逐包传输)，期间 PC 若发 0x50 START_SAMPLE 或发生器启动命令会被接受。虽然每次 ProgramBytes 仅 ~2ms 不构成 ISR 长阻塞，但 TIM6 采样 ISR 内做 I2C(有意设计)与主循环的 Flash Program 交织，且 STM32 在对自身 Flash 编程时取指会停顿(同一 Flash bank)，可能扰乱采样时序；逻辑上也允许在半写文件期间跑无关任务。建议：在 WriteBegin 成功后置一个协议层 file-store 会话标志，DispatchFrame 中比照 s_flashSessionActive 的做法，会话期内仅放行 flash-store 组+心跳，或至少拒绝 sampling/generator START。
建议：新增 s_fileStoreSessionActive(volatile)，WriteBegin 成功置位、WriteEnd/失败/Wipe 清零；在 DispatchFrame 中拒绝会话期内的 sampling/generator START 命令(或全部非 flash-store 组命令)。

低 低 (low) · 16 项

g_pmicHwenFlag 读-清-用非原子（理论上可丢一次边沿）
📦 Core (启动 / 中断 / 主循环框架层)  ·  📍 Core/Src/main.c:154-158  ·  🗳️ 验证 2/2 支持
问题：主循环 if(g_pmicHwenFlag){g_pmicHwenFlag=0; EnableSequence();}。若 EXTI4 在『读到非0』与『写0』之间又触发置1，则该次写0会清掉新边沿，丢失一次重配请求。flag 已正确声明 volatile，且 HWEN 边沿极低频、丢失最多延迟到下次真实边沿，实际影响可忽略，但严格讲是 ISR/主循环共享变量的 RMW 竞态。
建议：若需严格不丢边沿，可在 __disable_irq/__enable_irq 临界区内做读+清，或改用『先清后读硬件电平』模式。当前可保留并加注释说明可接受。
无独立看门狗（IWDG），死循环/总线死锁无法自恢复
📦 Core (启动 / 中断 / 主循环框架层)  ·  📍 Core/Src/main.c:142 (主循环) / 全局  ·  🗳️ 验证 2/2 支持
问题：主循环与各 Halt 死循环均无看门狗喂狗机制。若主循环被某个 Poll 内的意外阻塞卡住（如 I2C 异常路径），或 fault handler 进入死循环，系统只能靠人工断电恢复。对电机调试板这类需长期挂机的设备，缺少 IWDG 自恢复是健壮性隐患。CLAUDE.md TODO 已列为结构性待办。
建议：评估引入 IWDG：在 BSP 增加 bsp_iwdg，主循环每轮喂狗（Halt 死循环可故意不喂以触发复位或保留人工诊断窗口，需与诊断需求权衡）。属新增功能，按规约应先与开发者确认。
BSP_I2C_Scan / BSP_I2C2_Scan 无 pAddrList 上界保护
📦 BSP-comm (UART + I2C1/2/3)  ·  📍 BSP/Src/bsp_i2c.c:362-410; BSP/Src/bsp_i2c2.c:439-453  ·  🗳️ 验证 2/2 支持
问题：两个 Scan 都无条件 pAddrList[(*pCount)++]=addr，扫描范围 1..126 最多写 126 项。当前唯一调用方 app_protocol.c:363 用 addrList[127] 足够，但 API 签名未带容量参数，纯依赖调用方约定。任何容量 <126 的缓冲都会栈越界，且 CLAUDE.md TODO 已列『Scan pAddrList 无上界』为待修项。
建议：给 Scan 增加 maxCount 入参或在头文件用宏约定最小缓冲尺寸(如 BSP_I2C_SCAN_LIST_MIN=127)并在调用处静态断言；至少在写入前判断 *pCount < maxCount
RecoverBus 无法恢复从机钳死 SCL
📦 BSP-comm (UART + I2C1/2/3)  ·  📍 BSP/Src/bsp_i2c.c:183-234; BSP/Src/bsp_i2c2.c:457-488  ·  🗳️ 验证 2/2 支持
问题：恢复流程假定主机能自由驱动 SCL 做 9 脉冲。若从机一直把 SCL 拉低(钳死)，软件脉冲无效，恢复后 BUSY 仍置位（硬件路径 Init 会因此返 ERROR；软件路径 s_busRecovered 不会置位但无显式失败上报）。CLAUDE.md TODO 已记录此局限。
建议：恢复前/后读 SCL_READ，若 SCL 被钳低则直接判失败上报诊断码（类似 g_i2c2InitDiag），避免误判恢复成功
BSP_UART_Init / BSP_UART_SetBaudrate 恒返 SUCCESS，掩盖潜在硬件故障
📦 BSP-comm (UART + I2C1/2/3)  ·  📍 BSP/Src/bsp_uart.c:143-150,184-225  ·  🗳️ 验证 2/2 支持
问题：两函数无任何失败路径，即使 GPIO/DMA/USART 配置异常也返 SUCCESS。main.c 已对 Init 返回值做 HaltNoUart 处理(H-c)，但实际永远不会触发该分支；SetBaudrate 失败(如 TC 永不置位仅 break)也对上位机透明。CLAUDE.md 要求 BSP 初始化统一返回 ErrorStatus 以供上层判断。
建议：Init 可在末尾校验 USART_Cmd 后 USART->CR1 EN 位、SetBaudrate 校验 TC 排空是否超时(超时返 ERROR)，使返回值具备实际语义
SetEnable 非读改写，盲写固定值覆盖整寄存器
📦 BSP-misc (PMIC / TIM6 / Flash / LED / Tick)  ·  📍 BSP/Src/bsp_pmic.c:232  ·  🗳️ 验证 2/2 支持
问题：SetEnable 直接写固定 0x8C/0x0C 到 CTRL 寄存器，覆盖 Bit7 之外的所有位（模式位[3:2] 被硬编码为 0b11，其余位清 0）。若 RT5112WSC CTRL 寄存器还含 OCP/软启动/discharge 等配置位，或上电默认值非 0x0C，盲写会破坏这些设置；同时与 SetVout 配合时若器件要求保留某些位也会丢失。CLAUDE.md TODO 已将其列为低优先级待修项。
建议：改为读-改-写：ReadReg 取当前值，仅置/清 Bit7（及确需的模式位），其余位保留后写回；或在 .h 用掩码宏明确定义需保留位。
App_Generator_StartCosine API 自身无 channelCount 上界保护，存在栈/数组越界隐患
📦 App-Sample / App-Motor（示波器采样 + 波形发生器 + 电机 IC 寄存器访问）  ·  📍 App/Src/app_sample.c:611-629  ·  🗳️ 验证 2/2 支持
问题：s_cosAddr/s_cosPhaseX10 大小为 APP_GEN_COS_MAX_CH(3)，但 StartCosine 直接 for(i=0;i<channelCount;i++) 写入(:625-629)，未对 channelCount 做上界裁剪。当前唯一调用方 app_protocol.c:905-906 已校验 channelCount<=3，故现网安全；但 BSP/App 公开 API 不应依赖调用方校验（违反防御式编程，且测试代码/未来新调用方可越界写穿 s_cosAddr 后续静态变量）。
建议：在 StartCosine 入口加 if(channelCount==0U||channelCount>APP_GEN_COS_MAX_CH) return ERROR; 与 addrs/phaseX10 NULL 检查，使 API 自洽。
s_genMode 等发生器配置非原子更新，ISR 可能读到半套参数
📦 App-Sample / App-Motor（示波器采样 + 波形发生器 + 电机 IC 寄存器访问）  ·  📍 App/Src/app_sample.c:567-647  ·  🗳️ 验证 1/2 支持
问题：三个 Start 函数先置 s_genMode=GEN_NONE 再逐个写 s_linAddr/s_cos* 等配置，最后置 s_genMode=目标。但这些写之间不在临界区内，TIM6 ISR 若在配置写入过程中抢占：虽然 genMode 此刻为 GEN_NONE 不会执行写，问题较小；但 SetInterval(:484-494) 在 genMode 已=GEN_LINEAR/COSINE 时改 s_genDivider/s_cosDPhase，与 ISR 的 s_genCounter>=s_genDivider 判定(:242)及 Generator_DoCosineWrite 读 s_cosDPhase(:204) 并发，非 volatile 字段(s_cosDPhase/s_lin* 等)的多字节读写可能撕裂。
建议：对 ISR 与主循环共享的发生器配置(s_cosDPhase float、s_linStep 等)写入加 __disable_irq()/__enable_irq() 包裹，或在 Start/SetInterval 修改前临时 AcquireBus。至少 s_genDivider 已 volatile，s_cosDPhase 应同样处理。
App_Sample_SetChannelMask 未清 I2C 失败计数，且 mask 写入非临界区
📦 App-Sample / App-Motor（示波器采样 + 波形发生器 + 电机 IC 寄存器访问）  ·  📍 App/Src/app_sample.c:500-508  ·  🗳️ 验证 1/2 支持
问题：切换通道掩码后 s_i2cFailCount/s_i2cWarnSent 不复位，旧故障计数会延续到新掩码下，可能导致换通道后偏快触发降级/auto-stop。另 s_channelMask 是 volatile 单字节写入虽原子，但与 SetRegMap 的临界区不一致。CLAUDE.md TODO 低优先级已登记。
建议：SetChannelMask 成功路径内 __disable_irq 包裹并复位 s_i2cFailCount=0/s_i2cWarnSent=0，与寄存器映射变更行为对齐。
ReadData 在 maxLen < 剩余 chunk 时静默截断，调用方用更小 buffer 会造成静默丢数据
📦 App-FlashStore（片上 Flash 用户文件存储）  ·  📍 app_flashstore.c:272-277  ·  🗳️ 验证 2/2 支持
问题：chunkLen 先按 min(remaining, 252) 计算，再 if (chunkLen > maxLen) chunkLen = maxLen 二次钳位，且 actualLenOut 只报告钳位后的值、返回 FS_OK。由于 ReadData 的语义是 offset=pktSeq*FS_CHUNK_MAX_BYTES(固定 252 步进)，若某次调用 maxLen<252 导致读到 <252，下一包 pktSeq+1 的 offset 仍按 252 步进，会跳过 (252-actualLen) 个未读字节——表现为静默数据丢失而非报错。当前唯一调用方(app_protocol.c:1296)固定传 sizeof(buf)=252，所以现网安全，但这是一个脆弱的隐式契约。
建议：若 maxLen<FS_CHUNK_MAX_BYTES 且本包不是末包，应返回 FS_OUT_OF_RANGE 而非静默截断；或在 .h 契约中显式要求 maxLen>=FS_CHUNK_MAX_BYTES 并加 assert/早退。
jump_check 为永真 stub，烧录"成功"不代表固件可运行
📦 Flash-AW (AW86008/AW86100 ISP 烧录链)  ·  📍 Flash/AW/AW86008_86100/Src/aw_uboot_isp.c:276-326  ·  🗳️ 验证 2/2 支持
问题：aw_flash_jump_check 始终返回 ISP_OK，vendor 真实现被注释。aw_i2c_isp_download 返回 ISP_OK 只代表 erase+write 子步骤 ACK，并不代表 jump 成功或新固件运行。与路线图 4.6 "未做回读验证"一致，属已知风险但应用层/PC 端若把 ISP_OK 当作"烧录成功"会误判。
建议：已在代码注释明确，属可接受的已知 stub。建议：PC 端必须用 0x33 回读校验或 0x37 RESET + 可达性测试二次确认；联调拿到 vendor 真 ACK 格式后启用真实现。文档化即可，不改 vendor 代码。
pack_read 的 ACK 校验弱于 erase/write（仅查 r_buff[9]==0）
📦 Flash-AW (AW86008/AW86100 ISP 烧录链)  ·  📍 Flash/AW/AW86008_86100/Src/aw_uboot_isp.c:364-371  ·  🗳️ 验证 2/2 支持
问题：erase/write/hank 均用 aw_uboot_buff_built 重建期望 ACK 帧再 memcmp 比对(:453/508/607)，而 pack_read 仅检查 r_buff[9]==0(一个状态字节)，不校验地址/长度/校验和是否匹配。回读到错位或部分损坏的数据不会被发现，启用 0x33 回读校验时可能给出假阳性。
建议：启用回读校验前，建议(在 vendor 注释框内记录后)补一致性检查：至少校验返回帧的 module/event/校验和，或对 r_buff 头部做 memcmp，与 erase/write 路径对齐。属只审范畴，留待联调启用回读时处理。
block_write 循环计数器 i 为 uint8_t，len>63 时溢出死循环
📦 Flash-AW (AW86008/AW86100 ISP 烧录链)  ·  📍 Flash/AW/AW86008_86100/Src/aw_uboot_isp.c:465,475  ·  🗳️ 验证 2/2 支持
问题：aw_flash_block_write_ckeck 中 i 为 uint8_t，循环 for(i=0;i<len*4;i++)。当前所有调用 len≤16(i<64)安全；但若 vendor/未来调用传 len≥64，len*4≥256 截断为 uint8_t 后比较恒成立，i 回绕，造成 l_buff[i+4] 越界写(l_buff 仅 68)且死循环。属潜在脆弱点而非现触发 bug。
建议：调用方(download_check)硬编码块大小为 16，当前无法触发。建议在 download_check/aw_flash.h 文档中明确"单块 len 上限 16 字"的不变量约束；若担心可把 i 提升为 uint32_t(与其它函数一致，pack_read/download_check 的 i 已是 uint32_t)。
