通信协议 (App 协议层)

app_protocol.c（1636 行）实现帧状态机解析、CRC16 校验、命令分发与所有 handler。协议定义见 protocol.MD v1.4。

帧格式

控制帧（双向，可靠性优先）： [0xAA][0x55][SEQ][CMD][LEN][DATA n][CRC16_H][CRC16_L] CRC 范围 = SEQ+CMD+LEN+DATA（不含帧头），结果大端 数据流帧（STM32→PC，高频采样，只发不收）： [0xBB][MASK][LEN][DATA n][XOR] 每通道 2 字节 int16 大端，按通道号从低到高；XOR = MASK^LEN^所有数据
CRC16 注意：非标准 MODBUS
多项式 0x8005、初值 0xFFFF、LSB 先行（右移）、结果大端存放。注意这不是标准 CRC16-MODBUS（标准用反射多项式 0xA001）。PC 端实现需与此严格一致。

模块详情

协议应用层，是 PC 上位机与 STM32 之间唯一的命令通道。职责：(1) 用字节级状态机从 UART 环形缓冲解析控制帧 [0xAA][0x55][SEQ][CMD][LEN][DATA][CRC16_H][CRC16_L]；(2) CRC16(poly=0x8005, init=0xFFFF, 右移/LSB, 结果大端) 校验；(3) 通过 DispatchFrame 将 36 个命令码 (0x00~0x58) 分发到各 handler；(4) 统一的发送路径 (SendFrame/重试/ErrorResp/DebugInfo/BootStatus/FlashExecProgress)。同时承载 AW ISP 烧录 session (0x32~0x37) 与 STM32 片内 Flash 文件存储 (0x39~0x3F) 的状态机与并发互斥逻辑。不直接碰硬件寄存器，所有 IO 经 BSP/App_Motor/App_Sample/App_FlashStore。

公开 API

签名说明
void App_Protocol_Init(void)复位 RX 状态机及 bulk/flash session 标志，main 初始化最后一步调用。
void App_Protocol_Poll(void)主循环每轮调用：检查 I2C2 恢复标志与 UART 溢出计数(发 DebugInfo)，读环形缓冲全部字节逐字节喂 ProcessByte。
void App_Protocol_SendDebugInfo(const char *msg)发 0x06 ASCII 调试帧，SEQ=0xFF。msg 经 strlen 截成 uint8_t 长度。
void App_Protocol_SendErrorResp(uint8_t seq, Proto_ErrCode_t err)发 0x01 错误响应，1 字节错误码。
void App_Protocol_SendBootStatus(Proto_BootStatus_t status)发 0x0B 启动状态，SEQ=0xFF；可在 Init 前调用(仅依赖 BSP_UART)。
void App_Protocol_SendFlashExecProgress(uint8_t phase, uint32_t done, uint32_t total)发 0x38 烧录进度帧，SEQ=0xFF，LEN=9 全 LE；供 AW ISP on_progress 回调，丢帧容忍。

关键实现要点

RX 状态机 SOF 重同步

ProcessByte (app_protocol.c:1515)：8 态机。STATE_WAIT_SOF2 收到 0xAA 时停留在 SOF2(处理 0xAA 0xAA 0x55)，否则回 SOF1。LEN==0 直接跳 CRC_H。无帧超时机制——半截帧后若静默,状态机会停在中途态(见 findings)。

CRC16 实现

CRC16_Update (app_protocol.c:126) 右移变体，init 0xFFFF，每字节异或后循环 8 次。注意非标准 MODBUS(注释已明示 poly 用 0x8005 非反射 0xA001)。RX 累加从 SEQ 起(1544)，TX 同范围(167-176)，结果大端存放(179-180)/比较(1577)，与 protocol.MD 一致。

命令分发前置互斥闸

DispatchFrame (app_protocol.c:1361)：①s_bulkReadActive 时只回心跳;②s_flashSessionActive(仅 AW ISP 0x32 置位) 时只放行 flash 组+心跳,其余回 EXEC_FAIL;③采样运行时拒绝 0x30/0x31 透传(长事务不能 mask ISR)。

I2C2 单次事务总线锁

0x20/0x21/0x30/0x31 用 App_Sample_AcquireBus/ReleaseBus 包裹(471/504/649/708)，防 TIM6 采样 ISR 抢占软件 bit-bang 中途(H-d 已闭环)。0x22 bulk read 为长事务,改在入口拒绝 sample+generator(561)。

Bulk read 多包组帧

HandleBulkRead (app_protocol.c:518)：startReg+totalRegs*2 防 uint16 溢出(549)，BULK_MAX_TOTAL_REGS=32130 上界，ceil 分包，每包 [pktIdx][totalPkts][data]，静态 s_pktBuf。reg 地址 = startReg + idx*2(16-bit 字寻址)。

AW ISP 烧录 staging

64KB s_fwBuf(SRAM1,占 57%)。BEGIN 校验 addr∈[0x01000000,0x01020000) 且 totalBytes 4 对齐≤64KB(1022)。DATA 按 pktSeq 严格递增累加+越界检查(1063)。EXEC 入口拒绝 sample/generator/bulkread 并发(1102)，阻塞 ~5-10s。

片内 Flash 文件存储

0x39~0x3F 委托 App_FlashStore_*。0x39 BEGIN/0x3F WIPE 入口与 AW session/sample/generator/bulkread 互斥(1178/1318)。0x3A WriteData 用 uint16 chunkLen，0x3D ReadData 用 252B 栈缓冲。

输入长度边界校验

各 handler 对 pFrame->len 做精确/最小校验。0x30 写透传(606)：先 len>=3,再 len>=3+addrSize,再算 dataLen 并要求 len==3+addrSize+dataLen,逐级防 data[] 越界。0x56 cos: APP_GEN_COS_MAX_CH=3 → 7+3*4=19 无 uint8 溢出。

数据流 / 控制流

流程
RX：USART1 RXNE ISR → BSP 环形缓冲(256B) → 主循环 App_Protocol_Poll() → BSP_UART_RxRead 批量取到 s_rxBuf → 逐字节 ProcessByte 状态机累加 CRC → CRC_L 态比对：失败发 0x01(SEQ=0xFF)/成功 DispatchFrame → 命令 handler → 经 App_Motor/App_Sample/App_FlashStore/BSP_I2Cx 执行 → SendFrame 回 ACK/数据 或 SendErrorResp。 TX：SendFrame 在静态 s_txBuf 组帧(SOF+SEQ+CMD+LEN+DATA+大端CRC) → BSP_UART_TxWait → BSP_UART_Transmit(DMA)。SendFrameWithRetry 在 BSP busy 时 TxWait 后重试 1 次,再失败静默丢(依赖 PC 100ms 超时×2 重发)。控制流互斥：bulk/flash session/采样三类闸门在 DispatchFrame 顶部短路。

模块耦合关系

向下依赖：bsp_uart(收发/波特率/溢出计数)、bsp_i2c1/2/3(Scan/Probe/Transparent RW/恢复标志)、bsp_pmic、bsp_tim(GetPeriodUs)、app_motor(ReadReg/WriteReg)、app_sample(IsActive/AcquireBus/ReleaseBus/GetEffectiveMask/SetRegMap/Start/Stop 及 App_Generator_*)、app_flashstore、aw_flash(AW_86008_86100_Flash_Run/aw_reset_chip/ISP_STATUS_E)。向上：被 main 主循环与 AW ISP on_progress 回调调用。是 BSP 与上位机间的唯一协议网关，命令码枚举即对外 ABI。

本模块审查发现

严重 Flash 文件存储写 session(0x39→0x3B)期间未阻止采样/发生器启动 → I2C2 与片内 Flash 编程并发
📍 app_protocol.c:1361 (DispatchFrame) / 1168 HandleFlashStoreWriteBegin / 1201 HandleFlashStoreWriteData
问题：s_flashSessionActive 只由 AW ISP 0x32 BEGIN 置位(1036)，0x39 flashstore BEGIN 不置任何 session 闸门，仅在 BEGIN/WIPE 入口一次性检查 sample/generator 未运行。0x39 之后、0x3B END 之前，DispatchFrame 仍放行 0x50 START_SAMPLE 与 0x55/0x56/0x58 生成器启动。一旦采样启动，TIM6 ISR 驱动 I2C2 软件 bit-bang，而 0x3A WriteData 调 App_FlashStore_WriteData 编程片内 Flash——STM32F4 Flash 编程/擦除会阻塞 Flash 总线，从 Flash 取指的 TIM6 ISR 在此期间无法执行,破坏采样时序甚至取指异常。属真实并发缺口,非已记录的 H-e。
建议：为 flashstore 写 session 增加独立 active 标志(如 s_storeSessionActive)，在 0x39 成功后置位、0x3B/错误清零，并在 DispatchFrame 顶部像 AW session 一样拒绝该期间的 sample/generator/0x30/0x31/bulk 命令;或在 0x50/0x55/0x56/0x58 入口追加 flashstore-session 拒绝判断。
中 RX 状态机无半截帧超时,错误 LEN 会吞掉后续整帧
📍 app_protocol.c:1515 ProcessByte (STATE_WAIT_DATA 1561)
问题：若线路噪声/丢字节导致进入 STATE_WAIT_DATA 后期望的 len 字节迟迟收不齐，状态机无限期停留，期间到来的真正 0xAA 0x55 帧头被当作 DATA 吞掉，直到凑满 len 字节才进入 CRC 比对(必失败发 0x01)。一个被破坏的 LEN(如 0xFF)会连吃 255+2 字节，吞掉多个后续合法帧。PC 端 100ms 超时重发可缓解但会放大单次误码的影响面。
建议：引入帧内字节间超时(用 BSP_GetTick 记录上次收字节时刻,Poll 中若状态≠WAIT_SOF1 且超时则复位 s_state=STATE_WAIT_SOF1);或保留 SOF 双字节再同步作为兜底已部分覆盖,但 DATA 态无法借此恢复。
低 s_bulkReadActive 缺 volatile,被多上下文读取
📍 app_protocol.c:83 / 读取点 1104(EXEC 守卫) 1181 1321 1364
问题：s_bulkReadActive 在 HandleBulkRead 内置/清(主循环上下文)，但 DispatchFrame、HandleFlashExec/WriteBegin/Wipe 也读它做并发守卫。虽当前全部在主循环单线程上下文(bulk 期间 dispatch 只放行心跳,不会重入其他 handler)，无真正并发写,但编译器在 -O2 下可能将其缓存。已在 CLAUDE.md TODO 低优先级登记。
建议：补 volatile uint8_t s_bulkReadActive，与 s_flashSessionActive 一致,消除编译器优化隐患并明确语义。
低 HandleI2CScan 响应缓冲与协议上界一致但缺 count 上界断言
📍 app_protocol.c:361 HandleI2CScan
问题：respBuf[1+127]，count 来自 BSP_I2Cx_Scan(&count)。若某 BSP Scan 实现 bug 返回 count>127，则 respBuf[1+i] 与发送长度 1+count 越界(SendFrame 入参 len 为 uint8_t,1+128=129 仍合法但读越界 addrList)。当前 BSP 受 7-bit 地址天然限 127,实际安全,但依赖被调方契约。
建议：在 result==SUCCESS 后加 if (count>127) count=127; 防御性裁剪,或断言;与 CLAUDE.md TODO 中 'Scan pAddrList 无上界' 呼应。
风格 长度/参数错误统一返回 EXEC_FAIL(0x03)而非更细错误码
📍 全部 handler 的 len 校验分支
问题：protocol.MD/CLAUDE.md 仅定义 0x01 CRC/0x02 未知命令/0x03 执行失败三种错误码。所有长度不符、参数越界、范围错误均归并到 0x03,PC 端无法区分'帧结构错误'与'真实执行失败(如 I2C NACK)'。属协议设计取舍,不是 bug,但调试期定位困难。
建议：如后续协议扩展,可考虑新增 0x04 参数错误码;当前阶段保持现状,在 PcTest 用例中以命令上下文区分即可。
风格 DispatchFrame 巨型 switch,与 TODO 中'查找表'结构优化项一致
📍 app_protocol.c:1395-1508
问题：36 路 case 全为 'Handler(pFrame); break' 模板,新增命令需同步改枚举+switch,易遗漏。CLAUDE.md TODO 结构项已记 'DispatchFrame 查找表'。
建议：可改为 {cmd, fn_ptr} 静态查找表 + 前置闸门保留;非功能缺陷,优先级低。
