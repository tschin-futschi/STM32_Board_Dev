Flash 烧录 / 片上存储

两套独立的 Flash 子系统：① 把固件烧进 外部 AW86008/AW86100 电机芯片（经 I2C2 ISP 协议）；② 把用户文件存进 STM32 自身片上 Flash（Sector 5-11，917KB，CRC32 校验）。

① AW 芯片 ISP 烧录链 (Flash/AW/)

供应商移植代码
Flash/AW/AW86008_86100/ 是 AW 官方 ISP 代码移植到 STM32 I2C2 回调的产物。改动须在文件头加修改注释。2026-06-01 已修 H-a（ISP I2C 返回值检查）/ H-b（aw_flash_pack_read 栈缓冲越界 69→78）。
PC: 0x32 BEGIN [addr][totalBytes≤64KB] → 0x33 DATA [pktSeq][chunk] （分包灌入 STM32 staging buffer） → 0x34 EXEC （阻塞 5-10s: reset_chip→boot_control →block_erase→isp_download→jump_check） ↑ 期间 0x38 PROGRESS 帧上报 erase/write 进度 → resp [ispStatus]

AW86008/AW86100 电机 IC 的本地 ISP 烧录子系统，是供应商 SDK 的移植集成版。职责：通过 I2C2（软件 bit-banging）按 AW UBOOT 协议把固件写入目标芯片 Flash 并校验。分四层：aw_flash.c 顶层一站式入口（boot→download→reset）、aw_uboot_isp.c 协议事务实现（hank/erase/write/read/jump）、aw_uboot_bufbuilt.c 协议帧组装、aw_port_stm32.c 把 vendor 回调签名翻译到本项目 BSP（I2C2 透传 + 0x38 进度上报）。改动策略为只审不改的供应商代码，所有本地修改在文件头注释中按时间追加。EXEC 端到端阻塞约 5-13s，期间主循环停转。

关键实现要点

回调架构（移植关键）

aw_uboot_isp.c:72 s_ops 静态指针；95/102/111 三个 static 转发器(delay_ms/aw_i2c_write/aw_i2c_read)带 NULL 防御；aw_port_stm32.c:60 g_awOpsStm32 把 AW (DevId,AddrSize,pAddr,Size,Data)/int 翻译为 BSP (devAddr,pAddr,addrSize,pData,dataLen)/ErrorStatus。长度 uint8_t→uint16_t 隐式提升无损

两个 I2C 地址

aw_uboot_isp.c:66/70 AW_UBOOT_I2C_ADDR=0x6F(stay-in-uboot 协议帧) 与 AW_CHIP_I2C_ADDR=0x6B(上电 wake 阶段)，同一物理 I2C2 总线。两者均为推断值，注释标注实测不对需改

H-b 缓冲区修复（已确认彻底）

aw_flash_pack_read 读长度 = u32ReadLen*4+14，u32ReadLen=16 时=78；数据拷贝最大 r_buff[14+63]=r_buff[77]。r_buff/w_buff 均已=78(:332-333)，越界已消除。aw_flash_block_write_ckeck l_buff[68]/w_buff[77] 对 len≤16 也吻合(:467-468)

H-a I2C 返回值检查（已确认彻底）

AwWakeSequence(:135-160)、hank(:591/597)、erase(:437/443)、write(:488/498)、pack_read(:353/359) 全部检查返回值快速失败，跳过无谓 delay_ms。配合 S3(bsp_i2c2 超时/NACK 可返 ERROR)。aw_i2c_stop_uboot 的 20× 序列按 vendor 原意 fire-and-forget 不检查(:125)

阻塞延时（CLAUDE.md 允许，非 ISR）

AwPort_DelayMs(:26) 用 BSP_GetTick 忙等。write 每块 delay_ms(10)(:496)，erase delay_ms(25*len)(:441)。32KB(~512块)累计数秒。全在主循环上下文非 ISR，符合规约。EXEC 期间主循环停转，PC 端靠 0x38 进度帧 + START 提示扩大心跳窗口

进度回调(0x38)

download_check:536/546 erase 前后各一次(total=1)，:556/566 每写块成功后一次(total=divisor+remaind)。三处均有 (s_ops!=NULL && on_progress!=NULL) 双重 NULL 防御。on_progress=NULL 时 vendor 行为零差异

jump_check 是 stub

aw_flash_jump_check 永远返回 ISP_OK(:325)，vendor 真实现注释保留(:286-324)。意味 download 返 ISP_OK 仅代表 erase+write 子步骤 ACK，不代表新固件已运行——与路线图 4.6 备注一致，须 PC 端回读二次确认

并发保护(EXEC 入口)

app_protocol.c:1102 HandleFlashExec 入口拒绝 App_Sample_IsActive/App_Generator_IsRunning/s_bulkReadActive，避免 I2C2 并发。staging buffer s_fwBuf=64KB(FLASH_BUF_SIZE)，EXEC 传 s_fwTotalBytes/4 作 len(字)

数据流 / 控制流

流程
PC 经协议 0x32(INIT 设 addr/totalBytes)→0x33(DATA 分块填 s_fwBuf)→0x34(EXEC)。HandleFlashExec 入口互斥检查通过后调 AW_86008_86100_Flash_Run(s_fwAddr, s_fwBuf, s_fwTotalBytes/4)。Flash_Run 内：aw_boot_control(AwWakeSequence 经 0x6B wake + 20×0xAC 到 0x6F stay-in-uboot) → aw_i2c_isp_download(空间/NULL 检查 → hank_connect_check 握手 → flash_download_check 整块 erase + 16字/块 write 循环，每块经 aw_uboot_buff_built 组帧后 TransparentWrite + delay + TransparentRead + memcmp 校验，并触发 0x38 进度帧 → jump_check stub) → aw_reset_chip 让芯片运行新固件。所有 I2C 经 s_ops 转发器 → g_awOpsStm32 → BSP_I2C2_Transparent*(软件 bit-banging)。进度经 AwPort_OnFlashProgress → App_Protocol_SendFlashExecProgress(0x38 主动帧 UART)。

模块耦合关系

下游强耦合 BSP_I2C2_TransparentWrite/Read(软件 I2C2)、BSP_GetTick(忙等延时)；上游耦合 App_Protocol(0x32~0x38 命令、SendFlashExecProgress、s_fwBuf staging)。回调表经 aw_isp_init 在 main.c 注册解耦 vendor 与 BSP。aw_uboot_isp.c 依赖 aw_uboot_bufbuilt.c 组帧 + string.h(memcpy/memcmp)。与 app_sample/app_generator 通过 EXEC 入口互斥而非锁(因 EXEC 是长事务，不能 mask ISR 数秒)。include main.h(经 aw_uboot_isp.h)引入 CMSIS 类型。

本模块审查发现

严重 jump_check 为永真 stub，烧录"成功"不代表固件可运行
📍 Flash/AW/AW86008_86100/Src/aw_uboot_isp.c:276-326
问题：aw_flash_jump_check 始终返回 ISP_OK，vendor 真实现被注释。aw_i2c_isp_download 返回 ISP_OK 只代表 erase+write 子步骤 ACK，并不代表 jump 成功或新固件运行。与路线图 4.6 "未做回读验证"一致，属已知风险但应用层/PC 端若把 ISP_OK 当作"烧录成功"会误判。
建议：已在代码注释明确，属可接受的已知 stub。建议：PC 端必须用 0x33 回读校验或 0x37 RESET + 可达性测试二次确认；联调拿到 vendor 真 ACK 格式后启用真实现。文档化即可，不改 vendor 代码。
中 pack_read 的 ACK 校验弱于 erase/write（仅查 r_buff[9]==0）
📍 Flash/AW/AW86008_86100/Src/aw_uboot_isp.c:364-371
问题：erase/write/hank 均用 aw_uboot_buff_built 重建期望 ACK 帧再 memcmp 比对(:453/508/607)，而 pack_read 仅检查 r_buff[9]==0(一个状态字节)，不校验地址/长度/校验和是否匹配。回读到错位或部分损坏的数据不会被发现，启用 0x33 回读校验时可能给出假阳性。
建议：启用回读校验前，建议(在 vendor 注释框内记录后)补一致性检查：至少校验返回帧的 module/event/校验和，或对 r_buff 头部做 memcmp，与 erase/write 路径对齐。属只审范畴，留待联调启用回读时处理。
中 block_write 循环计数器 i 为 uint8_t，len>63 时溢出死循环
📍 Flash/AW/AW86008_86100/Src/aw_uboot_isp.c:465,475
问题：aw_flash_block_write_ckeck 中 i 为 uint8_t，循环 for(i=0;i<len*4;i++)。当前所有调用 len≤16(i<64)安全；但若 vendor/未来调用传 len≥64，len*4≥256 截断为 uint8_t 后比较恒成立，i 回绕，造成 l_buff[i+4] 越界写(l_buff 仅 68)且死循环。属潜在脆弱点而非现触发 bug。
建议：调用方(download_check)硬编码块大小为 16，当前无法触发。建议在 download_check/aw_flash.h 文档中明确"单块 len 上限 16 字"的不变量约束；若担心可把 i 提升为 uint32_t(与其它函数一致，pack_read/download_check 的 i 已是 uint32_t)。
中 两个 I2C 从机地址 0x6F/0x6B 均为推断值，未经实测确认
📍 Flash/AW/AW86008_86100/Src/aw_uboot_isp.c:66,70
问题：AW_UBOOT_I2C_ADDR=0x6F 与 AW_CHIP_I2C_ADDR=0x6B 注释自承"推断自序列""如实测不对改这里"。地址错则整个 wake/握手在 hank_connect_check 即失败返 ISP_HANK_ERROR。属联调阻塞项而非代码缺陷。
建议：联调首步用逻辑分析仪/I2C 抓包确认这两个地址；确认后更新注释去掉"推断"措辞。无需现在改。
低 AwWakeSequence 使用复合字面量数组作 I2C 入参（C99 局部生命周期）
📍 Flash/AW/AW86008_86100/Src/aw_uboot_isp.c:135-160
问题：形如 (uint8_t[]){0xff,0xa3,0x02} 复合字面量，其生命周期为所在块。当前 aw_i2c_write→BSP_I2C2_TransparentWrite 是同步阻塞软件 I2C(在调用内完整发完字节)，安全。但若回调实现改为异步/DMA(指针被延迟使用)，悬垂指针风险。当前实现安全。
建议：保持现状即可，因 I2C2 是软件同步发送。若未来 I2C 透传改异步需复审。可在注释中标注"依赖 i2c_write 同步语义"。
低 reset/boot 后未验证芯片实际状态，纯发命令 + 固定延时
📍 Flash/AW/AW86008_86100/Src/aw_uboot_isp.c:165-193
问题：aw_reset_chip/aw_boot_control 在 AwWakeSequence 成功后仅 delay_ms 收尾即返 ISP_OK；20×stop_uboot 为 fire-and-forget 不检查返回(按 vendor 原意)。无法确认芯片真进入/退出 uboot。与 jump stub 同源——成功返回信息量有限。
建议：已知 vendor 行为，文档化即可。回读校验启用后端到端确认。不改 vendor 代码。
风格 pack_read 中 tmp 初值 -1 赋给 uint8_t（隐式截断为 0xFF）
📍 Flash/AW/AW86008_86100/Src/aw_uboot_isp.c:335
问题：uint8_t tmp = -1; 依赖隐式截断为 0xFF 表达"未成功"，配合 r_buff[9]==0 时置 0。逻辑正确但 -Wconversion 下可能告警，且语义晦涩(vendor 笔法)。
建议：vendor 原始风格，不改。如未来重写可用 uint8_t tmp = 0xFFU 或布尔语义。

② 片上 Flash 文件存储 (app_flashstore.c)

单插槽覆盖模式
区域 Sector 5-11 [0x08020000, 0x08100000)，元数据 16B + 数据 917488B。写流程 BEGIN(整区擦)→DATA(顺序 pktSeq)→END(CRC32 校验通过才写元数据)。固件本体限于前 128KB（链接脚本防呆），与存储区物理隔离。

在 STM32 内部 Flash 的 Sector 5-11（896KB，[0x08020000, 0x08100000)）上实现单插槽文件存储。前 16B 为元数据（magic/size/crc32/reserved，小端），其后为最多 917488B 的连续文件数据。采用"会话写"模型：WriteBegin 整区擦除并初始化会话，WriteData 按 pktSeq 顺序逐包写入并累计 CRC32，WriteEnd 校验 PC 端 CRC 通过后才提交元数据，从而保证中断/失败的写入永不暴露半成品文件。读取无状态（ReadBegin 读元数据，ReadData 按包号偏移读）。CRC32 为 IEEE 802.3 表驱动，须与 PC 端逐字节一致。对接协议 0x39~0x3F 命令。

关键实现要点

CRC32 表驱动算法

app_flashstore.c:34-57。poly 0xEDB88320(IEEE 802.3 反射多项式)，运行时懒生成 256 项表(s_crcTable，1KB .bss)，init 0xFFFFFFFF，update 用 s_crcTable[(crc^byte)&0xFF]^(crc>>8)，最终 ~accum 取反。注释强调须与 PC firmware_parser.cpp crc32Compute 逐字节一致

单插槽会话状态机

5 个静态变量(app_flashstore.c:65-69): s_writeActive/s_expectedTotal/s_writeOffset/s_nextPktSeq/s_writeCrcAccum。WriteBegin 擦成功后置位，WriteData 推进，WriteEnd/写失败/Wipe 复位。注意这些不带 volatile，仅在主循环 poll(单线程)访问，符合设计

元数据原子提交

app_flashstore.c:184-217。WriteEnd 仅在 offset==total 且 CRC 匹配后才写 16B 元数据(magic=0xA5C3E18F)。WriteBegin 已擦 Sector5，元数据落在全 0xFF 区，保证 program 干净。中途崩溃/CRC 失败→无 magic→下次 Init/ReadBegin 报 FS_EMPTY，从不暴露半成品

容量与顺序边界

WriteBegin 校验 [1,917488]；WriteData 校验 pktSeq 严格递增(155-157)+len∈[1,252](158)+(uint32)offset+len<=total(161,显式提升避免溢出)。收到重复/跳号 pktSeq 返回 FS_SEQ_ERROR(不去重，与 CLAUDE.md 重发策略一致)

pktSeq→offset 读偏移

流程
写流程：PC 发 0x39(totalBytes,4B LE)→HandleFlashStoreWriteBegin 校验互斥+发 BEGIN 调试串+擦区+开会话→应答 1B FsStatus。随后 PC 循环发 0x3B(应为 0x3A) WRITE_DATA(pktSeq 2B LE + 数据≤252B)→handler 取 pktSeq、chunkLen=len-2→App_FlashStore_WriteData 顺序校验+ProgramBytes+CRC 累计→应答 nextSeq(2B)。全部发完 PC 发 0x3B WRITE_END(expectedCrc 4B LE)→比对 ~accum==expected→写元数据→应答 1B。读流程：0x3C READ_BEGIN→应答 status+size+crc32(9B)；0x3D READ_DATA(pktSeq 2B)→栈 buf[252]→ReadData memcpy→应答 actualLen 字节。0x3E INFO 应答 total+used(8B)；0x3F WIPE 同 BEGIN 互斥后整区擦。控制/数据均经主循环 App_Protocol_Poll 单线程串行处理。

中 文件写会话(WriteData/WriteEnd)无协议层并发互斥，sampling/generator 可在写入中途启动
📍 app_protocol.c:1200-1249 / app_flashstore.c:147-217
问题：0x39 WRITE_BEGIN 和 0x3F WIPE 在入口拒绝 sampling/generator/bulkRead 并发(因 3-7s 擦区阻塞)，但 0x3A WRITE_DATA / 0x3B WRITE_END 没有任何互斥检查。一个文件写会话(s_writeActive=1)可能持续数秒到数十秒(逐包传输)，期间 PC 若发 0x50 START_SAMPLE 或发生器启动命令会被接受。虽然每次 ProgramBytes 仅 ~2ms 不构成 ISR 长阻塞，但 TIM6 采样 ISR 内做 I2C(有意设计)与主循环的 Flash Program 交织，且 STM32 在对自身 Flash 编程时取指会停顿(同一 Flash bank)，可能扰乱采样时序；逻辑上也允许在半写文件期间跑无关任务。建议：在 WriteBegin 成功后置一个协议层 file-store 会话标志，DispatchFrame 中比照 s_flashSessionActive 的做法，会话期内仅放行 flash-store 组+心跳，或至少拒绝 sampling/generator START。
建议：新增 s_fileStoreSessionActive(volatile)，WriteBegin 成功置位、WriteEnd/失败/Wipe 清零；在 DispatchFrame 中拒绝会话期内的 sampling/generator START 命令(或全部非 flash-store 组命令)。
低 ReadData 在 maxLen < 剩余 chunk 时静默截断，调用方用更小 buffer 会造成静默丢数据
📍 app_flashstore.c:272-277
问题：chunkLen 先按 min(remaining, 252) 计算，再 if (chunkLen > maxLen) chunkLen = maxLen 二次钳位，且 actualLenOut 只报告钳位后的值、返回 FS_OK。由于 ReadData 的语义是 offset=pktSeq*FS_CHUNK_MAX_BYTES(固定 252 步进)，若某次调用 maxLen<252 导致读到 <252，下一包 pktSeq+1 的 offset 仍按 252 步进，会跳过 (252-actualLen) 个未读字节——表现为静默数据丢失而非报错。当前唯一调用方(app_protocol.c:1296)固定传 sizeof(buf)=252，所以现网安全，但这是一个脆弱的隐式契约。
建议：若 maxLen<FS_CHUNK_MAX_BYTES 且本包不是末包，应返回 FS_OUT_OF_RANGE 而非静默截断；或在 .h 契约中显式要求 maxLen>=FS_CHUNK_MAX_BYTES 并加 assert/早退。
