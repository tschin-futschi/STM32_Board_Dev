I2C2 上电不通讯事故复盘 高优先级 / 已解决

日期：2026-05-22  |  仓库：STM32_Board_Dev  |  修复 PR：#1 (4551285)  |  分支：fix/i2c2-power-on
TL;DR：固件上电后所有上位机指令全 timeout、心跳无响应。逐级 LED 闪烁定位到 BSP_I2C2_Init 失败；再细分到 SDA 和 SCL 同时被拉低；最终查明根因是 BSP_PMIC_Init 只预设电压寄存器、没有打开 LDO 输出，电机 IC 的 IOVDD 始终 = 0 V，I2C2 上拉无电压。修复：开机时显式调用 BSP_PMIC_EnableSequence()，并把 I2C2 初始化挪到 PMIC 上电之后。

一、故障现象

烧录最新固件（8a638e4 起的累积改动）后上电，上位机连接 COM7：

[motordev.serial] Port COM7 opened successfully
[motordev.config] SetPmicVoltage TX ...
[motordev.serial] Retry #1 for seq=0X00 cmd=0X09(SetPmicVoltage)
[motordev.serial] Retry #2 for seq=0X00 cmd=0X09(SetPmicVoltage)
[motordev.serial] Command timeout: seq=0X00 after 3 attempts
[motordev.serial] Heartbeat lost: 3 missed, disconnecting
  • 所有控制帧（SetPmicVoltage / I2cBusScan / 心跳）全部 timeout
  • LED 常灭，未进入主循环 5 Hz 心跳
  • 等同于固件完全死在初始化阶段，但具体死在哪一步从外部看不出来

二、定位过程（四阶段递进）

阶段 1 init 模块级二分

main.c 顶部临时插入 DiagBlinkN(N) 助手 + 11 个 checkpoint。每个 BSP/App 初始化成功后 LED 闪 N 次再继续，失败处保留 while(1) {;} halt。

static void DiagBlinkN(uint8_t n) {
    for (uint8_t i = 0; i < n; i++) {
        BSP_LED1_Toggle();
        for (volatile uint32_t k = 0; k < 1500000U; k++) {;}  // 150 ms on
        BSP_LED1_Toggle();
        for (volatile uint32_t k = 0; k < 1500000U; k++) {;}  // 150 ms off
    }
    for (volatile uint32_t k = 0; k < 10000000U; k++) {;}  // 1 s group separator
}

实测看到 ▮ ▮ ▮ — pause — [常灭]

最后一组 = 3 闪（I2C1 完成）→ 卡在 BSP_I2C2_Init 失败的 while(1)
阶段 2 函数级二分

BSP_I2C2_Init 唯一 ERROR 出口在 BSP_I2C2_RecoverBus() 之后 SDA 仍读到 0。新增 g_i2c2InitDiag 全局变量，在 return 前根据 SCL 电平区分两种失败模式：

if (!SDA_READ()) {
    BSP_I2C2_RecoverBus();
    if (!SDA_READ()) {
        g_i2c2InitDiag = SCL_READ() ? 2U : 1U;  // 1=两条都低, 2=只 SDA 低
        return ERROR;
    }
}

main.c 失败处改为 while(1) { DiagBlinkN(g_i2c2InitDiag); } 持续显示码值。

实测看到 ▮ ▮ ▮ — pause — ▮ — ▮ — ▮ ...

反复 1 闪 = SCL 也读到 0 → SDA 和 SCL 同时被拉低。不是 slave 抢 SDA，而是两条上拉根本没电压
阶段 3 假设 1：上拉接 PMIC IOVDD

I2C2 GPIO 配置为 Open-Drain + NoPull，高电平完全依赖外部上拉电阻。若上拉电阻接电机 IC 的 IOVDD（由 PMIC 供电），则在 PMIC 起来之前两条线都拉不高。

尝试：把 main.c init 顺序由 I2C1 → I2C2 → I2C3 → PMIC_Init 改为 I2C1 → I2C3 → PMIC_Init → PMIC_HwenInit → I2C2，让 PMIC 先初始化。

仍然 6 闪后反复 1 闪，I2C2 还是失败 → 假设方向对（PMIC 顺序确实该在前）但还差一步
阶段 4 根因锁定

查 bsp_pmic.c 发现关键事实：

ErrorStatus BSP_PMIC_Init(void) {
    /* Pre-set output voltages (LDO2=DRVVDD, LDO1=IOVDD, LDO3=VCMVDD) */
    if (BSP_PMIC_SetVout(BSP_PMIC_CH_LDO2, drvV) != SUCCESS) return ERROR;
    if (BSP_PMIC_SetVout(BSP_PMIC_CH_LDO1, ioV)  != SUCCESS) return ERROR;
    if (BSP_PMIC_SetVout(BSP_PMIC_CH_LDO3, vcmV) != SUCCESS) return ERROR;
    return SUCCESS;  // ← 只配置电压寄存器，未写使能位！
}

真正打开 LDO 输出的是 BSP_PMIC_EnableSequence()（按 LDO2 → 10ms → LDO1 → 10ms → LDO3 时序使能）。但它只在主循环检测到 HWEN 上升沿时被调用——开机阶段从来没人调它。

根因：BSP_PMIC_Init 命名 ≠ 行为。LDO 输出从未打开 → IOVDD = 0 V → I2C2 上拉无电压 → SDA/SCL 同时读 0 → init 失败

三、修复方案

在 main() 中显式调用 BSP_PMIC_EnableSequence()，强制开机时就把 LDO 打开。同时调整 init 顺序，让 I2C2 在 IOVDD 稳定后再做总线检查。

修复前（main 分支）

NVIC → Tick → LED → UART
→ I2C1_Init
→ I2C2_Init   ← 此时 IOVDD = 0 V，必失败
→ I2C3_Init
→ PMIC_Init  // 只配寄存器
→ HwenInit
→ aw_isp_init
→ App init → main loop
   // 主循环里 HWEN 中断才会触发
   // BSP_PMIC_EnableSequence()

修复后（fix/i2c2-power-on）

NVIC → Tick → LED → UART
→ I2C1_Init
→ I2C3_Init             // PMIC 总线
→ PMIC_Init             // 预设电压
→ PMIC_EnableSequence ← 开 LDO
→ HwenInit
→ I2C2_Init             // IOVDD 已上电
→ aw_isp_init
→ App init → main loop

3.2 保留的 HWEN 中断逻辑

开机时强制 EnableSequence 不影响原本的 HWEN 触发机制——主循环仍会监听 HWEN 上升沿并调用 BSP_PMIC_EnableSequence()，应对运行时 PMIC 被外部信号复位的场景。

五、根因分类与教训

5.1 直接根因

BSP_PMIC_Init 的命名暗示"初始化 PMIC"，但实际只设置电压寄存器、不开 LDO 输出。这种命名 ≠ 行为的 API 设计是事故的直接原因。

5.2 系统性原因

类别问题
API 设计"Init"职责不一致：I2C/UART/LED 的 Init 都能直接工作，唯独 PMIC_Init 还需要后续动作
启动时序把"硬件依赖关系"隐藏在主循环 HWEN 中断分支里，开机路径上没体现
错误处理回归旧版本曾用 (void)BSP_I2C2_Init() 旁路错误，看起来"能跑"但其实第一次访问总线就 NACK，问题被掩盖至此
诊断可见性while(1) {;} halt 对外部完全不可见。本次靠临时插 DiagBlinkN 才定位

5.3 改进建议

1. API 重命名：BSP_PMIC_Init → BSP_PMIC_PresetVout（或在 Init 内部完成 EnableSequence），让"Init = 可用"成为不变契约。
2. 启动失败可见化（已在 main 实现）：HaltOnInitFail(PROTO_INIT_FAIL_XXX) 通过 0x0B BOOT_STATUS 帧向上位机汇报具体失败模块 + LED 统一快闪。本次提交进一步加入了 PROTO_INIT_FAIL_PMIC_EN 码。
3. 现场调试入口保留：g_i2c2InitDiag 全局变量保留（无运行成本），未来再遇到 I2C2 总线问题可通过 JTAG / 调试器直接读出失败模式（1 = 双线低 / 2 = 仅 SDA 低）。
4. 严格 halt-on-fail，不再旁路：BSP Init 失败必须 halt（或 graceful degrade），禁止 (void)Init() 这种掩盖手法。

六、本次定位方法学要点

  1. 无串口时用 LED 编码：硬件唯一保证可用的输出口是 LED，DiagBlinkN(N) 是一种极简的"打印行号"机制，连续闪 N 次 + 长暗代表"通过第 N 个 checkpoint"
  2. 从模块级到函数级二分：先用 11 个 checkpoint 锁定哪个 init 失败，再用 g_i2c2InitDiag 区分函数内具体路径。这种递进二分比一次性放 50 个 checkpoint 效率更高
  3. 假设要可证伪：阶段 3 调整 init 顺序后失败模式不变，立刻意识到"PMIC 顺序对，但还缺一步"——单变量验证让方向修正成本最低
  4. 调试代码清理后保留长效价值：临时 DiagBlinkN 在修复后删除；但 g_i2c2InitDiag 作为低成本诊断指针保留下来
