系统架构 (Core 框架层)

Core 是固件骨架：上电启动、向量表、时钟、NVIC 分组、外设初始化序列与主循环调度，全部 ISR 集中于 stm32f4xx_it.c。本身不含业务逻辑。

启动序列

上电 → Reset_Handler (.s) 拷贝 .data / 清 .bss → SystemInit (配 180MHz 时钟, VTOR=0x08000000) → __libc_init_array → main(): NVIC_PriorityGroup_4 ← 必须第一行 BSP_Tick_Init (SysTick 1ms) BSP_LED_Init BSP_UART_Init ✗→ HaltNoUart (100ms 快闪, 无串口) BSP_I2C1_Init ✗→ HaltOnInitFail(I2C1) BSP_I2C3_Init ✗→ ... (PMIC 总线, 必须先于 I2C2) BSP_PMIC_Init ✗→ 预设电压 BSP_PMIC_EnableSequence✗→ 开 LDO, 给电机 IC 上拉供电 BSP_PMIC_HwenInit (EXTI4 武装) BSP_I2C2_Init ✗→ 电机 IC 总线 (上拉已上电) aw_isp_init ✗→ 注册 ISP 回调 App_Protocol/Sample/FlashStore_Init → 发 PROTO_BOOT_OK → 进主循环
关键依赖：PMIC 必须先于 I2C2
电机 IC 的 SDA/SCL 上拉电阻由 PMIC 的 IOVDD 供电，因此初始化顺序严格为 I2C3 → PMIC_Init → PMIC_EnableSequence → I2C2。颠倒会导致 I2C2 总线无上拉而初始化失败。

主循环

while (1) {
    now = BSP_GetTick();
    App_Protocol_Poll();        // 解析 UART 环形缓冲里的帧并响应
    App_Sample_Poll();          // 组帧经 DMA 发 0xBB 数据流
    if (g_pmicHwenFlag) {       // HWEN 上升沿 → 重跑 PMIC 使能
        g_pmicHwenFlag = 0;
        BSP_PMIC_EnableSequence();
    }
    if (now - prevTick >= 500)  // 心跳 LED
        BSP_LED1_Toggle();
}
双通道故障指示
UART 可用时：发 0x0B BOOT_STATUS 帧（6 种 PROTO_INIT_FAIL_* 区分模块）+ 500ms 慢闪；UART 自身故障：退化为 100ms 快闪，现场可目视区分。

模块详情

Core 层是固件的框架骨架，负责上电启动、向量表、时钟、NVIC 分组、外设初始化序列与主循环调度，以及全部中断服务程序（ISR 集中于 stm32f4xx_it.c）。它本身不含业务逻辑，按依赖关系依次拉起 BSP/App，并在初始化失败时通过串口 0x0B BOOT_STATUS 帧 + LED 闪烁双通道指示故障。主循环以非阻塞轮询方式驱动协议解析、采样发送、PMIC HWEN 重配与 LED 心跳。内存布局通过链接脚本约束：固件仅占前 128KB Flash，余 896KB 留给 Flash 文件存储；堆为 0，栈 2KB，禁止动态分配。

数据流 / 控制流

流程
上电→Reset_Handler(.s:65) 拷贝.data/清.bss→SystemInit（配 180MHz 时钟、VTOR）→__libc_init_array→main()。main 内串行初始化各外设，任一返 !=SUCCESS 即跳 Halt 死循环。初始化完成后进主循环：每轮 BSP_GetTick 取时刻→App_Protocol_Poll（解析 UART 环形缓冲里的帧并响应）→App_Sample_Poll（组帧经 DMA 发 0xBB 数据流）→检查 g_pmicHwenFlag 决定是否重跑 PMIC 使能→按 500ms 翻转心跳 LED。中断侧：USART1 RX 填缓冲，TIM6 触发采样置标志/采集，DMA2_S7 完成发送回调，EXTI4 置 HWEN 标志，SysTick 累加毫秒。ISR 与主循环经 volatile 标志（g_pmicHwenFlag、s_sampleFlag、s_txDone、g_tick）单向通信。

模块耦合关系

Core 是顶层编排者，向下强依赖 BSP 全部模块（tick/led/uart/i2c1/i2c2/i2c3/pmic/tim）与 App 层（protocol/sample/flashstore）及 vendor AW 端口（aw_port_stm32 的 g_awOpsStm32 + aw_isp_init）。stm32f4xx_it.c 依赖 BSP 回调（BSP_UART_RxISR_Callback、BSP_UART_TxDmaISR_Callback、BSP_SampleTim_ISR_Callback、BSP_Tick_Increment）与 g_pmicHwenFlag。向量表符号名（TIM6_DAC_IRQHandler 等）与 startup .s 弱别名硬绑定。所有引脚/优先级宏来自 BSP/Inc，Core 不裸写端口（HardFault_Handler 除外，见 findings）。

为何 I2C2 用软件 bit-banging？
电机 IC 需要最高 1 MHz 的 I2C 时钟，而 STM32F4 硬件 I2C 最高只能到 400 kHz，因此 I2C2 改用基于 DWT->CYCCNT 周期计数延时的软件位翻转实现，并支持 16-bit 寄存器地址（大端）。

BSP 通信驱动 — UART / I2C1 / I2C2 / I2C3

BSP 通信驱动层，封装 USART1（环形缓冲 RX 中断 + DMA TX）与三路 I2C 总线。I2C1（INA）/I2C3（PMIC+GYRO）为硬件 I2C，经 bsp_i2c.c 公共实例化驱动（轮询 + 软件超时 + RecoverBus）复用；I2C2（电机 IC）为独立的软件 bit-banging 实现，用 DWT->CYCCNT 做半周期延时以达到 1 MHz，并支持 16-bit 大端寄存器地址与 AW Firmware 透传读写。该层只做字节级收发与总线恢复，不含协议/业务逻辑；并发保护（NVIC mask）由 App 层在 I2C2 调用外侧完成。

关键实现要点

UART RX 环形缓冲溢出处理

bsp_uart.c:261-275 RXNE ISR 计算 nextHead=(head+1)%512，nextHead==tail 时丢弃新字节并 s_rxOverflowCount++（符合 CLAUDE.md『溢出丢弃新数据』）。head 仅 ISR 写、tail 仅主循环写，单生产者单消费者无锁安全

DMA TX 单缓冲忙判断

bsp_uart.c:160-181 先查 s_txDone==0 返 BUSY，否则置 0、memcpy、按 RM0090 §10.3.17 五步重启流（DISABLE→等 EN 清→清 5 个 flag→设 M0AR/NDTR→ENABLE）。wait_dma_disable_or_recover() 10ms 超时则 DeInit+重初始化兜底。TC ISR 仅置 s_txDone=1

软件 I2C2 时序

bsp_i2c2.h:33 HALF_CYCLE=180MHz/1MHz/2=90 周期(500ns)；DELAY_LOW 扣减 SCL_LOW_ADJ=6 补偿 GPIO 操作开销。bit 原语用 DWT->CYCCNT 计时，BSRR 直接置位、IDR 读线。WaitSclHigh 支持从机时钟拉伸，STRETCH_TIMEOUT=36000 周期(200us)

s_busTimeout 拉伸守卫

bsp_i2c2.c:36 每次事务入口清 0，任一 WaitSclHigh 超时即置 1；SUCCESS 返回前(WriteReg:292/ReadReg:330/Transparent:370,416)检查，避免把拉伸期间读到的脏数据当真实返回（修复历史 S3）。注释明确非 volatile 因事务不跨 ISR/main 边界

16-bit 大端寄存器地址

bsp_i2c.c:103-119(硬件)与 bsp_i2c2.c:283-284/316-317(软件)均先发 reg>>8 再发 reg&0xFF。reg 参数全程 uint16_t，i2c1/i2c3 封装用 (uint16_t)reg 上提 8-bit，符合 CLAUDE.md 禁止强转 uint8_t

总线恢复 RecoverBus

硬件(bsp_i2c.c:183)：软件复位+引脚转 GPIO OD 拉高+9 个 SCL 脉冲+手动 STOP+恢复 AF 重初始化+可选回调。软件(bsp_i2c2.c:457)：9 个 SCL 脉冲+手动 STOP，成功置 s_busRecovered。两者均无法解决从机把 SCL 钳死的场景（已在 CLAUDE.md TODO 记录）

硬件 I2C 单字节读时序

bsp_i2c.c:296-316 N=1 路径：等 ADDR→先 DISABLE ACK→读 SR1/SR2 清 ADDR→GenerateSTOP→等 RXNE→读 1 字节，符合 STM32F4 参考手册单字节接收规范

数据流 / 控制流

流程
RX：USART1 RXNE 中断→stm32f4xx_it.c:83→BSP_UART_RxISR_Callback 写环形缓冲→主循环 App_Protocol_Poll 调 BSP_UART_RxRead 取出解析。TX：App 层组帧→BSP_UART_Transmit→memcpy s_txBuf(SRAM1)→DMA2 Stream7 搬运→TC 中断→stm32f4xx_it.c:92→TxDmaISR 置 s_txDone=1。I2C：App handler→(I2C2 由 App 层 Acquire/ReleaseBus NVIC mask 包裹)→BSP_I2Cx_*→硬件经 bsp_i2c.c 公共实现，软件 I2C2 经 bit-bang 原语；任一超时/NACK 触发 RecoverBus 并向上返 ERROR，App 层据此回 NACK（BSP 不本地重试）。

模块耦合关系

依赖 SPL（GPIO/USART/DMA/I2C/RCC/NVIC）与 CMSIS（DWT/CoreDebug）；bsp_uart.c 依赖 bsp_tick.c 的 BSP_GetTick 做超时。被 App 层调用：app_protocol.c（控制帧收发、Scan、寄存器读写、AW 透传）、app_sample.c（0xBB 数据流 Transmit）。I2C2 并发保护逻辑位于 App 层（app_protocol.c/app_sample.c 的 AcquireBus/ReleaseBus），BSP 自身不做 NVIC mask。stm32f4xx_it.c 提供 ISR 入口转调 BSP 回调。g_motorIcAddr/g_motorIcName 全局由协议层设置。

低 s_busRecovered 在软件 I2C2 恢复成功后置位但事务仍返 ERROR，标志读取与事务结果不联动

关键实现要点

PMIC 上电时序

EnableSequence (bsp_pmic.c:86) 固定顺序 LDO2(DRVVDD 1.8V)→10ms→LDO1(IOVDD 2.8V)→10ms→LDO3(VCMVDD 3.2V)，每步用 WaitMs 阻塞延时（基于 BSP_GetTick）；末尾三次回读校验 Bit7。Disable 为逆序。LDO4 永久禁用。

电压换算

SetVout (bsp_pmic.c:211) regVal=(uint8_t)((voltV-0.6)*80+0.5f)，含 +0.5 四舍五入；范围检查用 !(>=min && <=max) 同时拦截 NaN。步进 12.5mV。

HWEN 中断

EXTI4_IRQHandler (stm32f4xx_it.c:106) 上升沿触发，清 pending 后再读 PE4 电平做简单去毛刺，置 g_pmicHwenFlag=1；main.c:154 主循环消费该标志并调 EnableSequence。ISR 不做 I2C，符合规约。

TIM6 频率换算

PSC=89 使 90MHz/90=1MHz 计数时钟，ARR=interval_us-1。k_arrTable 7 档 (bsp_tim.c:17)，与 .h 注释一致。SetFreq 改 ARR 后 GenerateEvent(Update) 立即装载并清 IT pending，避免立即误触发。

TIM6 ISR

TIM6_DAC_IRQHandler 清 IT 后调 ISR_Callback；回调内先执行 s_callback（按 memory 是有意在 ISR 内做 I2C 采样以保证时序）再置 s_sampleFlag=1。s_sampleFlag/s_callback 均 volatile。

Flash 地址保护

ProgramBytes (bsp_flash.c:61) 拒绝 addr<0x08020000 / addr>=0x08100000 / len>END-addr（用减法防溢出），再校验 4 字节对齐；EraseSectors 校验 start>=5 && start<=end && end<=11。固件扇区 0~4 受保护。

Flash 扇区映射

s_sectorMap[12] 将平面索引 0~11 映射到 SPL 的 FLASH_Sector_x 宏（非线性 SNB 编码）；每次操作自带 Unlock/ClearFlag/Lock 配对。

数据流 / 控制流

流程
控制流：main 初始化序列里 LED→...→PMIC(Init+HwenInit)→采样定时器。运行期：(1) SysTick 每 1ms 触发 BSP_Tick_Increment，全系统时间基准；(2) PC 指令经协议层调用 BSP_PMIC_SetVout/SetEnable/EnableSequence 等读写 PMIC（I2C3）；(3) 硬件 HWEN 上升沿→EXTI4 ISR 置 g_pmicHwenFlag→主循环异步触发 EnableSequence 重新上电；(4) 采样开启后 TIM6 周期中断→ISR 内回调采样+置 s_sampleFlag→主循环 App_Sample_Poll 消费；(5) Flash 擦写由 App_FlashStore 调用，长时间阻塞 CPU（单 bank）。PMIC 依赖 bsp_i2c3 做实际字节收发，依赖 bsp_tick 做延时。

模块耦合关系

bsp_pmic 依赖 bsp_i2c3（WriteReg/ReadReg）和 bsp_tick（WaitMs）；EXTI4 ISR 与 main.c 通过 g_pmicHwenFlag 单向耦合。bsp_tim 与 stm32f4xx_it.c（TIM6_DAC_IRQHandler）及 App_Sample（回调+标志）耦合。bsp_tick 被几乎所有需要超时/延时的模块依赖（I2C 超时、PMIC 延时、主循环心跳）。bsp_flash 被 App 层 flashstore/AW ISP 调用，独立无外设依赖。bsp_led 仅被 main 心跳调用。所有驱动经 .h 宏隔离硬件参数。

通信协议 (App 协议层)

app_protocol.c（1636 行）实现帧状态机解析、CRC16 校验、命令分发与所有 handler。协议定义见 protocol.MD v1.4。

帧格式

控制帧（双向，可靠性优先）： [0xAA][0x55][SEQ][CMD][LEN][DATA n][CRC16_H][CRC16_L] CRC 范围 = SEQ+CMD+LEN+DATA（不含帧头），结果大端 数据流帧（STM32→PC，高频采样，只发不收）： [0xBB][MASK][LEN][DATA n][XOR] 每通道 2 字节 int16 大端，按通道号从低到高；XOR = MASK^LEN^所有数据
CRC16 注意：非标准 MODBUS
多项式 0x8005、初值 0xFFFF、LSB 先行（右移）、结果大端存放。注意这不是标准 CRC16-MODBUS（标准用反射多项式 0xA001）。PC 端实现需与此严格一致。

关键实现要点

RX 状态机 SOF 重同步

ProcessByte (app_protocol.c:1515)：8 态机。STATE_WAIT_SOF2 收到 0xAA 时停留在 SOF2(处理 0xAA 0xAA 0x55)，否则回 SOF1。LEN==0 直接跳 CRC_H。无帧超时机制——半截帧后若静默,状态机会停在中途态(见 findings)。

CRC16 实现

CRC16_Update (app_protocol.c:126) 右移变体，init 0xFFFF，每字节异或后循环 8 次。注意非标准 MODBUS(注释已明示 poly 用 0x8005 非反射 0xA001)。RX 累加从 SEQ 起(1544)，TX 同范围(167-176)，结果大端存放(179-180)/比较(1577)，与 protocol.MD 一致。

命令分发前置互斥闸

DispatchFrame (app_protocol.c:1361)：①s_bulkReadActive 时只回心跳;②s_flashSessionActive(仅 AW ISP 0x32 置位) 时只放行 flash 组+心跳,其余回 EXEC_FAIL;③采样运行时拒绝 0x30/0x31 透传(长事务不能 mask ISR)。

I2C2 单次事务总线锁

0x20/0x21/0x30/0x31 用 App_Sample_AcquireBus/ReleaseBus 包裹(471/504/649/708)，防 TIM6 采样 ISR 抢占软件 bit-bang 中途(H-d 已闭环)。0x22 bulk read 为长事务,改在入口拒绝 sample+generator(561)。

Bulk read 多包组帧

HandleBulkRead (app_protocol.c:518)：startReg+totalRegs*2 防 uint16 溢出(549)，BULK_MAX_TOTAL_REGS=32130 上界，ceil 分包，每包 [pktIdx][totalPkts][data]，静态 s_pktBuf。reg 地址 = startReg + idx*2(16-bit 字寻址)。

AW ISP 烧录 staging

64KB s_fwBuf(SRAM1,占 57%)。BEGIN 校验 addr∈[0x01000000,0x01020000) 且 totalBytes 4 对齐≤64KB(1022)。DATA 按 pktSeq 严格递增累加+越界检查(1063)。EXEC 入口拒绝 sample/generator/bulkread 并发(1102)，阻塞 ~5-10s。

片内 Flash 文件存储

0x39~0x3F 委托 App_FlashStore_*。0x39 BEGIN/0x3F WIPE 入口与 AW session/sample/generator/bulkread 互斥(1178/1318)。0x3A WriteData 用 uint16 chunkLen，0x3D ReadData 用 252B 栈缓冲。

输入长度边界校验

各 handler 对 pFrame->len 做精确/最小校验。0x30 写透传(606)：先 len>=3,再 len>=3+addrSize,再算 dataLen 并要求 len==3+addrSize+dataLen,逐级防 data[] 越界。0x56 cos: APP_GEN_COS_MAX_CH=3 → 7+3*4=19 无 uint8 溢出。

数据流 / 控制流

流程
RX：USART1 RXNE ISR → BSP 环形缓冲(256B) → 主循环 App_Protocol_Poll() → BSP_UART_RxRead 批量取到 s_rxBuf → 逐字节 ProcessByte 状态机累加 CRC → CRC_L 态比对：失败发 0x01(SEQ=0xFF)/成功 DispatchFrame → 命令 handler → 经 App_Motor/App_Sample/App_FlashStore/BSP_I2Cx 执行 → SendFrame 回 ACK/数据 或 SendErrorResp。 TX：SendFrame 在静态 s_txBuf 组帧(SOF+SEQ+CMD+LEN+DATA+大端CRC) → BSP_UART_TxWait → BSP_UART_Transmit(DMA)。SendFrameWithRetry 在 BSP busy 时 TxWait 后重试 1 次,再失败静默丢(依赖 PC 100ms 超时×2 重发)。控制流互斥：bulk/flash session/采样三类闸门在 DispatchFrame 顶部短路。

app_sample.c 是示波器采样与波形发生器双功能模块，二者共用 TIM6 作为统一节拍源：I2C 读（采样）和寄存器写（发生器）均在 TIM6 ISR 内执行以保证时序精度（有意设计），主循环只负责把双缓冲数据经 UART/DMA 发出。采样组帧为 0xBB 数据流帧（protocol.MD v1.4）。app_motor.c 是电机 IC 16-bit 大端寄存器访问的薄封装，转调 BSP_I2C2，并用 g_motorIcAddr==0 做未配置守卫。模块对外提供采样控制、通道掩码/寄存器映射配置、I2C2 总线锁（AcquireBus/ReleaseBus 通过屏蔽 TIM6 IRQ 实现）以及三种发生器（线性三角波/锯齿/余弦）。

关键实现要点

双缓冲 ISR→主循环交接

s_isrBuf[2][8] 乒乓缓冲。ISR 写 bufIdx=s_isrWriteIdx，写完元数据后 s_isrWriteIdx ^= 1 并置 s_isrDataReady=1（app_sample.c:280-288）。Poll 在临界区读 readIdx=s_isrWriteIdx^1 并清 ready（:433-439），随后在临界区外 SendStreamFrame 读该 buffer（:447）

有效掩码计算两处

ISR 内（:260-267）和 App_Sample_GetEffectiveMask（:547-561）逻辑完全重复：channelMask & 已映射(!=0xFFFF) 通道

余弦相位累加器与 2π wrap

k_twoPi=6.28318530f 局部 const（:184），s_cosPhaseAccum 每节拍 += s_cosDPhase，>=2π 时减 2π 防 float 精度丢失（:204-208）；相位增量 dPhase=2πf·Δt 在 Start/SetInterval 预算（:636,:493）

auto-stop / 告警守卫

连续全通道失败计数：>=10 发降级告警 0x06，>=50 发 EXEC_FAIL+调试串并 App_Sample_Stop()。已加 totalCount>0 守卫防空映射 0>=0 误触发（:342）

总线锁实现

AcquireBus 用 NVIC_DisableIRQ(TIM6_DAC_IRQn)+__DSB+__ISB（:534-536），仅屏蔽采样 ISR；挂起的 TIM6 update 在 Release 后塌缩为 1 次 ISR。被 0x20/0x21/0x30/0x31 单次事务包裹（app_protocol.c:471,504,649,708）

发生器入口边界由协议层把关

数据流 / 控制流

流程
采样：TIM6 ISR → SampleTimerISR：先做发生器写（按 divider 计数）→ 再按有效掩码逐通道 App_Motor_ReadReg(I2C2) → 写入 s_isrBuf[bufIdx][ch] → 更新 mask/fail/total 元数据 → 交换 writeIdx + 置 dataReady。主循环 App_Sample_Poll：TX 不忙且 dataReady → 临界区快照元数据与 readIdx + 清 ready → SendStreamFrame 从 s_isrBuf[readIdx] 组 0xBB 帧（mask/LEN/大端 int16/XOR）→ BSP_UART_Transmit(DMA)。发生器：ISR 内 Generator_DoWrite → 线性/锯齿/余弦计算值 → App_Motor_WriteReg(I2C2)。主循环单次寄存器事务用 AcquireBus(屏蔽 TIM6)→App_Motor_*→ReleaseBus 防与 ISR 采样争用 I2C2。

② 片上 Flash 文件存储 (app_flashstore.c)

单插槽覆盖模式
区域 Sector 5-11 [0x08020000, 0x08100000)，元数据 16B + 数据 917488B。写流程 BEGIN(整区擦)→DATA(顺序 pktSeq)→END(CRC32 校验通过才写元数据)。固件本体限于前 128KB（链接脚本防呆），与存储区物理隔离。

在 STM32 内部 Flash 的 Sector 5-11（896KB，[0x08020000, 0x08100000)）上实现单插槽文件存储。前 16B 为元数据（magic/size/crc32/reserved，小端），其后为最多 917488B 的连续文件数据。采用"会话写"模型：WriteBegin 整区擦除并初始化会话，WriteData 按 pktSeq 顺序逐包写入并累计 CRC32，WriteEnd 校验 PC 端 CRC 通过后才提交元数据，从而保证中断/失败的写入永不暴露半成品文件。读取无状态（ReadBegin 读元数据，ReadData 按包号偏移读）。CRC32 为 IEEE 802.3 表驱动，须与 PC 端逐字节一致。对接协议 0x39~0x3F 命令。

CRC32 表驱动算法

单插槽会话状态机

5 个静态变量(app_flashstore.c:65-69): s_writeActive/s_expectedTotal/s_writeOffset/s_nextPktSeq/s_writeCrcAccum。WriteBegin 擦成功后置位，WriteData 推进，WriteEnd/写失败/Wipe 复位。注意这些不带 volatile，仅在主循环 poll(单线程)访问，符合设计

元数据原子提交

app_flashstore.c:184-217。WriteEnd 仅在 offset==total 且 CRC 匹配后才写 16B 元数据(magic=0xA5C3E18F)。WriteBegin 已擦 Sector5，元数据落在全 0xFF 区，保证 program 干净。中途崩溃/CRC 失败→无 magic→下次 Init/ReadBegin 报 FS_EMPTY，从不暴露半成品

流程
写流程：PC 发 0x39(totalBytes,4B LE)→HandleFlashStoreWriteBegin 校验互斥+发 BEGIN 调试串+擦区+开会话→应答 1B FsStatus。随后 PC 循环发 0x3B(应为 0x3A) WRITE_DATA(pktSeq 2B LE + 数据≤252B)→handler 取 pktSeq、chunkLen=len-2→App_FlashStore_WriteData 顺序校验+ProgramBytes+CRC 累计→应答 nextSeq(2B)。全部发完 PC 发 0x3B WRITE_END(expectedCrc 4B LE)→比对 ~accum==expected→写元数据→应答 1B。读流程：0x3C READ_BEGIN→应答 status+size+crc32(9B)；0x3D READ_DATA(pktSeq 2B)→栈 buf[252]→ReadData memcpy→应答 actualLen 字节。0x3E INFO 应答 total+used(8B)；0x3F WIPE 同 BEGIN 互斥后整区擦。控制/数据均经主循环 App_Protocol_Poll 单线程串行处理。

📦 App-Protocol (app_protocol.c / app_protocol.h)  ·  📍 app_protocol.c:1515 ProcessByte (STATE_WAIT_DATA 1561)  ·  🗳️ 验证 2/2 支持
问题：若线路噪声/丢字节导致进入 STATE_WAIT_DATA 后期望的 len 字节迟迟收不齐，状态机无限期停留，期间到来的真正 0xAA 0x55 帧头被当作 DATA 吞掉，直到凑满 len 字节才进入 CRC 比对(必失败发 0x01)。一个被破坏的 LEN(如 0xFF)会连吃 255+2 字节，吞掉多个后续合法帧。PC 端 100ms 超时重发可缓解但会放大单次误码的影响面。

📦 App-Sample / App-Motor（示波器采样 + 波形发生器 + 电机 IC 寄存器访问）  ·  📍 App/Src/app_sample.c:432-447  ·  🗳️ 验证 2/2 支持
