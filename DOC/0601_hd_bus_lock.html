H-d：采样/发生器运行时 I²C2 事务的并发保护「补不全」

高优先 待修（≡ 历史 H1，架构级）

① 根因：软件 bit-bang 不可被打断

电机 IC 走 I²C2 软件 bit-banging —— 逐位翻转 GPIO + DWT 精确延时的事务。 有两处都会操作 I²C2：

谁何时用 I²C2
TIM6 采样 ISR（中断上下文）采样 / 信号发生器运行时，读/写电机寄存器（有意设计，为采样时序精度）
主循环寄存器 IO0x20/0x21/0x22/0x30/0x31 handler

② 冲突：事务做一半被 ISR 抢断 → 时序损坏

主循环正在发 bulk read 的一个 I²C2 字节（bit-bang 中）： 主循环: SCL↑ SDA=b6 SCL↓ SCL↑ SDA=b5 ┊╳━━━━━━━━━━━━━━╳┊ SCL↓ ...继续 b4 ▲ ▲ TIM6 ISR: └─ 抢断！ISR 也翻 SCL/SDA 读采样 ─┘ 结果：两个事务的电平翻转交错 → SDA/SCL 乱跳 → 读到错数据 / 总线挂死

③ 锁机制：屏蔽 TIM6 ISR

/* app_sample.c — Acquire = 屏蔽采样 ISR，Release = 恢复 */
App_Sample_AcquireBus();   // NVIC_DisableIRQ(TIM6_DAC_IRQn)
... I²C2 事务 ...          // 此间 ISR 不会抢断
App_Sample_ReleaseBus();   // NVIC_EnableIRQ(TIM6_DAC_IRQn)

④ 现状：只补了一半

/* 0x20 已包锁（正例） */
App_Sample_AcquireBus();
ErrorStatus s = App_Motor_ReadReg(reg, &val);
App_Sample_ReleaseBus();

/* 0x22 bulk read 循环 —— 无锁（H-d） */
for (pktIdx = 0; pktIdx < totalPkts; pktIdx++) {
    for (i = 0; i < regsThisPkt; i++) {
        App_Motor_ReadReg(...);   // ← 无 AcquireBus 保护
    }
    BSP_UART_TxWait();
    SendFrameWithRetry(...);
}

⑤ 当前两道弱防线 + 漏洞

bulk read 现在靠：① 入口 App_Sample_IsActive() 拒绝（:559）；② 置 s_bulkReadActive=1（:566）。

⚠ 漏洞：入口只挡「采样（0x50）」，不挡信号发生器（0x55/0x56/0x58） —— 发生器同样在 TIM6 ISR 里写 I²C2。发生器运行时来一条 bulk read，入口放行 → ISR 抢断 → 总线损坏。

⑥ 为什么没顺手修

不能把整个 bulk read 循环包进一对 AcquireBus/ReleaseBus —— 那会连续屏蔽采样 ISR 几百毫秒（255 包 × (I²C 读 + UART 发)），把采样饿死。

正确做法需按包/按事务粒度加锁：

  • 只锁每包的 I²C 读循环，放开 UART 发送间隙（让 ISR 能在包之间运行）
  • 入口补 App_Generator_IsRunning() 校验，堵住发生器漏洞
  • 0x30/0x31 透传同样按事务包锁

因需设计取舍，列为待修（CLAUDE.md ## TODO 高优先 H-d / H1）。
