采样示波器 / 波形发生器

app_sample.c（666 行）实现两类功能：① 周期性采样电机 IC 寄存器并以 0xBB 数据流上报（示波器）；② 向电机寄存器周期性写入线性/锯齿/余弦波形（信号发生器）。两者通过 TIM6 中断驱动，共享 I2C2 总线锁。

总线并发保护（已修 H-d / H1）
采样与发生器在 TIM6 ISR 内访问 I2C2；单次事务命令（0x20/0x21/0x30/0x31）用 App_Sample_AcquireBus / ReleaseBus（NVIC 屏蔽 TIM6 IRQ）包装；长事务（0x22 bulk read、Flash 烧录）改在入口拒绝采样+发生器运行，避免长时间屏蔽 ISR。
采样频率档位（TIM6, 1MHz 计数）
通过 SET_INTERVAL 索引选择，ARR = 1MHz/freq − 1。波形发生器支持线性扫描、锯齿（每 tick 步进）、余弦（amp/offset/freq + 多通道）。

模块详情

app_sample.c 是示波器采样与波形发生器双功能模块，二者共用 TIM6 作为统一节拍源：I2C 读（采样）和寄存器写（发生器）均在 TIM6 ISR 内执行以保证时序精度（有意设计），主循环只负责把双缓冲数据经 UART/DMA 发出。采样组帧为 0xBB 数据流帧（protocol.MD v1.4）。app_motor.c 是电机 IC 16-bit 大端寄存器访问的薄封装，转调 BSP_I2C2，并用 g_motorIcAddr==0 做未配置守卫。模块对外提供采样控制、通道掩码/寄存器映射配置、I2C2 总线锁（AcquireBus/ReleaseBus 通过屏蔽 TIM6 IRQ 实现）以及三种发生器（线性三角波/锯齿/余弦）。

关键实现要点

双缓冲 ISR→主循环交接

s_isrBuf[2][8] 乒乓缓冲。ISR 写 bufIdx=s_isrWriteIdx，写完元数据后 s_isrWriteIdx ^= 1 并置 s_isrDataReady=1（app_sample.c:280-288）。Poll 在临界区读 readIdx=s_isrWriteIdx^1 并清 ready（:433-439），随后在临界区外 SendStreamFrame 读该 buffer（:447）

有效掩码计算两处

ISR 内（:260-267）和 App_Sample_GetEffectiveMask（:547-561）逻辑完全重复：channelMask & 已映射(!=0xFFFF) 通道

余弦相位累加器与 2π wrap

k_twoPi=6.28318530f 局部 const（:184），s_cosPhaseAccum 每节拍 += s_cosDPhase，>=2π 时减 2π 防 float 精度丢失（:204-208）；相位增量 dPhase=2πf·Δt 在 Start/SetInterval 预算（:636,:493）

auto-stop / 告警守卫

连续全通道失败计数：>=10 发降级告警 0x06，>=50 发 EXEC_FAIL+调试串并 App_Sample_Stop()。已加 totalCount>0 守卫防空映射 0>=0 误触发（:342）

总线锁实现

AcquireBus 用 NVIC_DisableIRQ(TIM6_DAC_IRQn)+__DSB+__ISB（:534-536），仅屏蔽采样 ISR；挂起的 TIM6 update 在 Release 后塌缩为 1 次 ISR。被 0x20/0x21/0x30/0x31 单次事务包裹（app_protocol.c:471,504,649,708）

发生器入口边界由协议层把关

channelCount<=APP_GEN_COS_MAX_CH(3) 的检查在 app_protocol.c:905-906，App_Generator_StartCosine 自身的 for 循环（app_sample.c:625-629）无上界保护

电机寄存器大端

App_Motor_ReadReg buf[0]<<8|buf[1]，WriteReg buf[0]=高字节（app_motor.c:28,44），符合 CLAUDE.md 16-bit 大端规约

数据流 / 控制流

流程
采样：TIM6 ISR → SampleTimerISR：先做发生器写（按 divider 计数）→ 再按有效掩码逐通道 App_Motor_ReadReg(I2C2) → 写入 s_isrBuf[bufIdx][ch] → 更新 mask/fail/total 元数据 → 交换 writeIdx + 置 dataReady。主循环 App_Sample_Poll：TX 不忙且 dataReady → 临界区快照元数据与 readIdx + 清 ready → SendStreamFrame 从 s_isrBuf[readIdx] 组 0xBB 帧（mask/LEN/大端 int16/XOR）→ BSP_UART_Transmit(DMA)。发生器：ISR 内 Generator_DoWrite → 线性/锯齿/余弦计算值 → App_Motor_WriteReg(I2C2)。主循环单次寄存器事务用 AcquireBus(屏蔽 TIM6)→App_Motor_*→ReleaseBus 防与 ISR 采样争用 I2C2。

模块耦合关系

向下依赖 app_motor.c→bsp_i2c2（软件 bit-banging I2C2 + g_motorIcAddr/g_motorIcName）；依赖 bsp_tim（TIM6 Init/Start/Stop/SetFreq/GetPeriodUs/SetCallback）；依赖 bsp_uart（Transmit/IsTxBusy/DMA）；依赖 app_protocol（SendErrorResp/SendDebugInfo + 错误码宏）。被 app_protocol.c 命令分发调用（0x20/0x21/0x30/0x31 用总线锁，0x50~ 采样控制，发生器命令，0x54 SetRegMap）。与 app_protocol 的 s_bulkReadActive / IsActive / IsRunning 互锁实现 I2C2 并发保护。CCM 禁止 DMA，s_streamBuf/s_isrBuf 须在 SRAM1（默认 .bss 即 SRAM1，符合）。

本模块审查发现

中 H-e：双缓冲数据与元数据可能不同源（通道错位/数据撕裂）
📍 App/Src/app_sample.c:432-447
问题：Poll 在临界区(__disable_irq/__enable_irq, :433-439)只快照了 mask/failCnt/totalCnt/readIdx 并清 ready，但真正读取数据 buffer 的 SendStreamFrame(s_isrBuf[readIdx],...) 在 :447 临界区之外执行。SendStreamFrame 是纯内存拷贝+UART_Transmit 排队，本身较快，但若两次 TIM6 ISR 在此期间抢占（采样频率高/Poll 被其它 Poll 拖慢时可能），ISR 会再次写回 readIdx 这块 buffer（writeIdx 在两次翻转后回到 readIdx），导致 SendStreamFrame 读到的数据是新一帧的、而 mask/totalCount 仍是旧快照——极端时序下通道数据与掩码错位。这是 CLAUDE.md TODO 已登记的 H-e。
建议：两种修复其一：①在临界区内把 s_isrBuf[readIdx] 的有效字节一次性 memcpy 到本地/发送缓冲再出临界区组帧；②采用三缓冲或在 ISR 侧检测 readIdx 即将被覆盖时跳过(dataReady 仍为 1 表示主循环未取走则不翻转覆盖)。推荐方案①，拷贝量最大 16 字节，临界区开销可控。
中 App_Generator_StartCosine API 自身无 channelCount 上界保护，存在栈/数组越界隐患
📍 App/Src/app_sample.c:611-629
问题：s_cosAddr/s_cosPhaseX10 大小为 APP_GEN_COS_MAX_CH(3)，但 StartCosine 直接 for(i=0;i<channelCount;i++) 写入(:625-629)，未对 channelCount 做上界裁剪。当前唯一调用方 app_protocol.c:905-906 已校验 channelCount<=3，故现网安全；但 BSP/App 公开 API 不应依赖调用方校验（违反防御式编程，且测试代码/未来新调用方可越界写穿 s_cosAddr 后续静态变量）。
建议：在 StartCosine 入口加 if(channelCount==0U||channelCount>APP_GEN_COS_MAX_CH) return ERROR; 与 addrs/phaseX10 NULL 检查，使 API 自洽。
中 s_genMode 等发生器配置非原子更新，ISR 可能读到半套参数
📍 App/Src/app_sample.c:567-647
问题：三个 Start 函数先置 s_genMode=GEN_NONE 再逐个写 s_linAddr/s_cos* 等配置，最后置 s_genMode=目标。但这些写之间不在临界区内，TIM6 ISR 若在配置写入过程中抢占：虽然 genMode 此刻为 GEN_NONE 不会执行写，问题较小；但 SetInterval(:484-494) 在 genMode 已=GEN_LINEAR/COSINE 时改 s_genDivider/s_cosDPhase，与 ISR 的 s_genCounter>=s_genDivider 判定(:242)及 Generator_DoCosineWrite 读 s_cosDPhase(:204) 并发，非 volatile 字段(s_cosDPhase/s_lin* 等)的多字节读写可能撕裂。
建议：对 ISR 与主循环共享的发生器配置(s_cosDPhase float、s_linStep 等)写入加 __disable_irq()/__enable_irq() 包裹，或在 Start/SetInterval 修改前临时 AcquireBus。至少 s_genDivider 已 volatile，s_cosDPhase 应同样处理。
低 SampleTimerISR 内多通道 I2C 读为长阻塞操作，与协议层 mask-ISR 数 ms 风险对应
📍 App/Src/app_sample.c:270-282
问题：软件 bit-banging I2C2 在 ISR 内逐通道读，最多 8 通道，每次含超时等待；这是 CLAUDE.md 认可的有意设计(采样时序精度)，非缺陷本身。但需注意 AcquireBus 用 NVIC mask TIM6 时，若主循环长事务(0x22 bulk)未被入口拒绝，会 mask ISR 数 ms。当前 0x22 已在入口拒绝采样+发生器，风险已缓解，仅作记录。
建议：无需改动；保持 0x22 等长事务入口拒绝采样/发生器的策略，确保 AcquireBus 只用于单次短事务。
低 App_Sample_SetChannelMask 未清 I2C 失败计数，且 mask 写入非临界区
📍 App/Src/app_sample.c:500-508
问题：切换通道掩码后 s_i2cFailCount/s_i2cWarnSent 不复位，旧故障计数会延续到新掩码下，可能导致换通道后偏快触发降级/auto-stop。另 s_channelMask 是 volatile 单字节写入虽原子，但与 SetRegMap 的临界区不一致。CLAUDE.md TODO 低优先级已登记。
建议：SetChannelMask 成功路径内 __disable_irq 包裹并复位 s_i2cFailCount=0/s_i2cWarnSent=0，与寄存器映射变更行为对齐。
低 发生器写返回值全部 (void) 丢弃，I2C 写失败上位机无感知
📍 App/Src/app_sample.c:117,151,200
问题：Generator_DoLinearWrite/Sawtooth/Cosine 对 App_Motor_WriteReg 返回值一律 (void) 丢弃。发生器在 ISR 内运行不便直接回包，但若电机 IC 持续 NACK，上位机完全无法察觉写失败(不像采样有 fail 计数+auto-stop)。
建议：参照采样的失败计数机制，为发生器加一个连续写失败计数，超阈值发 SendDebugInfo 告警(类似 :354)，避免静默失败。
低 GetEffectiveMask 读 s_channelRegMap 非原子，与 SetRegMap 临界区写存在竞态
📍 App/Src/app_sample.c:547-561
问题：GetEffectiveMask 在主循环(App_Sample_Start 及协议层 CheckTickOverrun)调用，逐元素读 s_channelRegMap[ch]；而 SetRegMap(:513-519) 在 __disable_irq 临界区内逐元素写。两者均在主循环上下文(非 ISR)通常串行，但 GetEffectiveMask 未保证读到一致快照。影响小但与 ISR 内同样的掩码计算共三处重复。
建议：将有效掩码计算抽成单一 static helper，主循环调用处按需加临界区；消除 ISR(:260)/GetEffectiveMask(:552)/重复实现。
风格 Generator_DoLinearWrite 与 Generator_DoSawtoothWrite 三角波推进逻辑完全重复
📍 App/Src/app_sample.c:114-180
问题：两函数的‘升降三角波推进’代码逐字重复(:120-145 与 :154-179 一致)，仅写值前缀注释不同。维护时易只改一处导致行为分叉。
建议：抽取公共 static void AdvanceTriangle(void) 推进 s_linCurrent/s_linAscending，两个 DoWrite 各自只保留 App_Motor_WriteReg 调用。
