H-b：aw_flash_pack_read 栈缓冲区越界

严重度：栈破坏（越界写） 触发：仅 0x33 回读 / upload 路径

文件：Flash/AW/AW86008_86100/Src/aw_uboot_isp.c:313 — AW 芯片 flash 的「按包回读」函数，供 0x33 回读 / upload 校验使用。

① 问题：接收缓冲区开小了

/* :317-318 */
uint8_t w_buff[78] = { 0 };   // 发送缓冲，78 字节
uint8_t r_buff[69] = { 0 };   // 接收缓冲，只有 69 字节 ← 隐患

供应商原始代码两个缓冲尺寸不对称：发送给了 78，接收却只给 69。

② 越界点 A — 写越界（:342）

/* :342 */
(void)aw_i2c_read(AW_UBOOT_I2C_ADDR, 0U, NULL,
                  (uint8_t)(u32ReadLen * 4 + 1 + 4 + 9), r_buff);
//                读取长度 = u32ReadLen*4 + 14，直接透传给 BSP_I2C2_TransparentRead 写入 r_buff

aw_flash_read(:372-376）把数据按 16 字一块切分，每块以 u32ReadLen = 16 调用本函数。代入最大值：

读取长度 = 16 * 4 + 14 = 64 + 14 = 78 字节
r_buff 容量       = 69 字节
────────────────────────────
向 69 字节缓冲写入 78 字节 → 越界写 9 字节

这 9 字节会踩踏栈上相邻数据（相邻局部变量 / 返回地址附近），属栈破坏， 行为不可预测：可能 HardFault，也可能静默改写其他变量。

③ 越界点 B — 读越界（:353-354）

/* :353-354 */
for (i = 0; i < u32ReadLen * 4; i++) {            // i 最大到 63
    byDataBuff[u32PackNum * 64 + i] = r_buff[14 + i];   // 14 + 63 = 77
}
最大读取索引 = r_buff[14 + 63] = r_buff[77]，超出 69 的边界 → 越界读

④ 触发条件（为什么现在"还没炸"）

aw_i2c_isp_upload(:254) → aw_flash_read(:360) → aw_flash_pack_read(..., 16)

路径是否经过 pack_read结果
主烧录 EXEC（0x32 download）否当前烧录不触发
回读 / upload 校验（0x33）是（u32ReadLen=16）触发栈破坏

CLAUDE.md 路线图 4.6 记着「芯片可运行新固件未做回读验证」—— 一旦启用 0x33 回读来确认烧录成功，这个栈破坏立刻发生。 因此它属于「启用回读校验前必修」。

⑤ 修复

两个越界点的最大需求都是 78（读取长度 78、最大索引 77）。 把接收缓冲改到 78 即可：

uint8_t r_buff[69] = { 0 };   →   uint8_t r_buff[78] = { 0 };

因属供应商代码，按项目约定需在文件头补一条修改注释说明本次改动。
