I2C 驱动架构：bsp_i2c 与 bsp_i2c1/2/3 的交互关系

重构后的 I2C 驱动采用 实例配置结构体 + 公共驱动 + 薄包装层 的设计模式。本文解释三层之间的完整交互流程。

2. 核心机制：配置结构体

BSP_I2C_Config_t 结构体描述一条 I2C 总线的全部硬件参数。公共驱动的每个函数都以 const BSP_I2C_Config_t *cfg 作为第一个参数，通过 cfg-> 访问硬件寄存器，而不是硬编码宏。

typedef struct
{
    I2C_TypeDef    *periph;         /* I2C1 / I2C2 / I2C3         */
    uint32_t        rccApb1;        /* RCC 时钟使能位             */
    uint32_t        speed;          /* 400000U                    */
    uint32_t        timeout;        /* 超时循环计数               */

    /* SCL 引脚（独立字段，兼容 I2C3 双端口） */
    GPIO_TypeDef   *sclPort;
    uint16_t        sclPin;
    ...

    /* SDA 引脚 */
    GPIO_TypeDef   *sdaPort;
    uint16_t        sdaPin;
    ...

    uint8_t         regAddr16;      /* 0=8-bit寄存器, 1=16-bit寄存器 */
    BSP_I2C_PostRecoverFn pfnPostRecover; /* 恢复成功后的回调      */
} BSP_I2C_Config_t;

3. 薄包装层的工作方式

每个 bsp_i2cX.c 做两件事：

3.1 定义一个 static const 配置实例

用各自 bsp_i2cX.h 中的硬件宏填充结构体字段：

/* bsp_i2c1.c */
static const BSP_I2C_Config_t k_i2c1Cfg = {
    .periph       = BSP_I2C1_PERIPH,       /* → I2C1 */
    .sclPort      = BSP_I2C1_SCL_GPIO_PORT, /* → GPIOB */
    .sclPin       = BSP_I2C1_SCL_PIN,       /* → GPIO_Pin_6 */
    .sdaPort      = BSP_I2C1_SDA_GPIO_PORT, /* → GPIOB (同一端口) */
    .sdaPin       = BSP_I2C1_SDA_PIN,       /* → GPIO_Pin_7 */
    .regAddr16    = 0U,                     /* 8-bit 寄存器地址 */
    .pfnPostRecover = NULL,                 /* 无回调 */
};

/* bsp_i2c3.c — 注意 SCL/SDA 在不同 GPIO 端口 */
static const BSP_I2C_Config_t k_i2c3Cfg = {
    .sclPort      = BSP_I2C3_SCL_GPIO_PORT, /* → GPIOA */
    .sdaPort      = BSP_I2C3_SDA_GPIO_PORT, /* → GPIOC (不同端口!) */
    ...
};

/* bsp_i2c2.c — 16-bit 地址 + 恢复回调 */
static const BSP_I2C_Config_t k_i2c2Cfg = {
    .regAddr16      = 1U,                   /* 16-bit 寄存器地址 */
    .pfnPostRecover = PostRecoverCallback,   /* 恢复后置标志 */
    ...
};

3.2 提供一行转发函数

每个公开 API 函数只做一件事：把 &k_i2cXCfg 传给公共驱动：

ErrorStatus BSP_I2C1_WriteReg(uint8_t devAddr, uint8_t reg,
                               const uint8_t *pData, uint16_t len)
{
    return BSP_I2C_WriteReg(&k_i2c1Cfg, devAddr, (uint16_t)reg, pData, len);
}                            ▲
                             │
                    把配置实例的地址传进去
关键点：App 层调用 BSP_I2C1_WriteReg(addr, reg, data, len)，签名完全没变。薄包装层在中间加了一个 &k_i2c1Cfg 参数后转发给 BSP_I2C_WriteReg()。App 层代码零修改。

4. 公共驱动如何处理三个总线的差异

三条 I2C 总线有三个差异点，全部通过结构体字段自动处理：

差异 涉及总线 结构体字段 公共驱动中的处理
寄存器地址宽度 I2C2 = 16-bit
I2C1/I2C3 = 8-bit		 regAddr16 WritePhase() 中 if (cfg->regAddr16) 分支
SCL/SDA 端口 I2C3: GPIOA + GPIOC
I2C1/I2C2: 同一端口		 sclPort / sdaPort 统一用独立初始化（I2C1/I2C2 两个 port 值相同，无影响）
恢复后通知 仅 I2C2 有 pfnPostRecover RecoverBus() 末尾 if (callback != NULL) callback()

4.1 寄存器地址宽度（WritePhase 内部）

static ErrorStatus WritePhase(const BSP_I2C_Config_t *cfg,
                               uint8_t devAddr, uint16_t reg)
{
    ...
    if (cfg->regAddr16 != 0U)
    {
        /* I2C2 走这里：先发高字节，再发低字节 */
        I2C_SendData(cfg->periph, (uint8_t)(reg >> 8U));
        /* ...等待发送完成... */
        I2C_SendData(cfg->periph, (uint8_t)(reg & 0xFF));
        /* ...等待发送完成... */
    }
    else
    {
        /* I2C1 / I2C3 走这里：只发一个字节 */
        I2C_SendData(cfg->periph, (uint8_t)reg);
        /* ...等待发送完成... */
    }
}

4.2 GPIO 双端口（ConfigGpioAF 和 RecoverBus 内部）

static void ConfigGpioAF(const BSP_I2C_Config_t *cfg)
{
    /* SCL 和 SDA 分别初始化，不合并到一个 GPIO_Init 调用 */
    gpioInit.GPIO_Pin = cfg->sclPin;
    GPIO_Init(cfg->sclPort, &gpioInit);  /* I2C3: GPIOA */

    gpioInit.GPIO_Pin = cfg->sdaPin;
    GPIO_Init(cfg->sdaPort, &gpioInit);  /* I2C3: GPIOC (不同!) */
}
为什么不合并？旧的 I2C1/I2C2 代码用 GPIO_Pin_SCL | GPIO_Pin_SDA 一次初始化两个引脚，但这要求它们在同一个端口。I2C3 的 SCL(PA8) 和 SDA(PC9) 分属 GPIOA 和 GPIOC，必须分开调用 GPIO_Init()。统一用分别初始化的方式，对 I2C1/I2C2 无影响（两个 port 值相同），但完美兼容 I2C3。

4.3 恢复回调（RecoverBus 末尾）

void BSP_I2C_RecoverBus(const BSP_I2C_Config_t *cfg)
{
    /* ... 9 个 SCL 脉冲 + 手动 STOP + 重初始化 ... */

    /* 恢复成功后的回调 */
    if ((cfg->pfnPostRecover != NULL) &&
        (I2C_GetFlagStatus(cfg->periph, I2C_FLAG_BUSY) == RESET))
    {
        cfg->pfnPostRecover();  /* I2C2: 置 s_busRecovered = 1 */
    }
}

I2C2 的配置中 .pfnPostRecover = PostRecoverCallback，该回调定义在 bsp_i2c2.c 中：

/* bsp_i2c2.c */
static uint8_t s_busRecovered = 0U;

static void PostRecoverCallback(void)
{
    s_busRecovered = 1U;
}

I2C1 和 I2C3 的配置中 .pfnPostRecover = NULL，公共驱动检测到 NULL 就跳过回调。

7. 设计收益

  • Bug 修复只改一处：以前改一个 I2C bug 要同步修改三个文件，现在只改 bsp_i2c.c
  • 不会功能分叉：以前 I2C2 有 recovery flag 而 I2C1/I2C3 没有，是因为复制代码后忘记同步。现在通过回调机制统一处理
  • 代码量减少 55%：三份驱动共 1168 行 → 公共 310 行 + 包装 220 行 = 530 行
  • App 层零修改：所有 BSP_I2Cx_* 函数签名完全不变，调用者无感知
  • 新增 I2C 总线极简：只需新建一个包装文件，填写 BSP_I2C_Config_t 并写转发函数
