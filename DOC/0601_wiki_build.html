构建系统 / 测试框架

Windows MSYS2/MinGW 环境，arm-none-eabi-gcc + SPL V1.9.0 + OpenOCD/ST-Link。

make              # Release (-O2)
make DEBUG=1      # Debug (-Og -g3 -DDEBUG)
make flash        # 编译并烧录
make clean
关键编译标志
-mcpu=cortex-m4 -mfpu=fpv4-sp-d16 -mfloat-abi=hard -mthumb
-ffunction-sections -fdata-sections -fno-common · -Wall -Wextra -Wshadow
-Wl,--gc-sections -specs=nano.specs（死代码裁剪，测试宏=0 时自动剔除测试代码）
测试框架 (Test/)
test_config.h 每个测试项一个使能宏，默认 0；测试代码只调 BSP/App 公开 API，不改业务文件。PC 端测试脚本在 PcTest/，仅依赖 pyserial。

模块详情

本模块覆盖固件的构建系统（Makefile + 链接脚本）与测试基础设施（片上 Test/ 框架 + PC 端 pyserial 脚本）。Makefile 用 arm-none-eabi-gcc(MSYS2) 编译 SPL V1.9.0，统一 -O2/Release 与 -Og/Debug，链接脚本将固件区人为压缩到 128KB 以与后 896KB Flash 文件存储区物理隔离。Test/ 框架通过 test_config.h 使能宏（默认全 0）配合 --gc-sections 实现"测试代码零成本进正式固件"。PcTest/ 是上位机协议验证脚本，独立实现 CRC16/帧封装来与固件互测。实测 make DEBUG=1 编译通过，gc-sections 已正确裁剪禁用测试代码。

关键实现要点

编译标志齐全性(CLAUDE.md §3 核对)

Makefile:100/119/120/130-131 — CPU_FLAGS(-mcpu=cortex-m4 -mfpu=fpv4-sp-d16 -mfloat-abi=hard -mthumb)、-ffunction-sections -fdata-sections -fno-common、-Wall -Wextra -Wshadow、-Wl,--gc-sections -specs=nano.specs 全部齐全。预定义宏 USE_STDPERIPH_DRIVER/HSE_VALUE=8000000U 齐全

--gc-sections 死代码裁剪验证

实测 mingw32-make DEBUG=1 后 build/firmware.map 中 Test_PMIC_PidRead_Poll/Test_I2C_Scan_Run 出现 0 次(仅在 Discarded input sections 列表)，证明默认 test 宏=0 时测试代码被完全裁剪，符合 CLAUDE.md §9 '正式固件不引入测试代码'

内存分配

Linker .ld:23-24 _Min_Stack_Size=0x800(2KB)、_Min_Heap_Size=0x000(禁 malloc)，与 CLAUDE.md 一致。CCM 区在 MEMORY 声明但无任何 SECTION 落入(DMA 禁访 CCM 得到保证)。实测 text=29696 data=104 bss=71544，flash≈29.8KB<<128K，RAM≈70KB+栈<112K，余量充足

PC↔固件协议一致性

4 个 PcTest 脚本的 CRC16 全部 poly=0x8005 右移大端，与固件 app_protocol.c:126 CRC16_Update 一致；帧格式 0xAA55+SEQ+CMD+LEN+DATA+CRC 一致；16-bit 寄存器用 struct '>H'/'>HH' 大端打包(i2c_command:429/439/451)，符合 CLAUDE.md '寄存器地址 16-bit 大端'。命令码 0x20/0x21/0x22/0x30/0x31/0x09 与 app_protocol.h 枚举一致

链接脚本 Flash 隔离设计

FLASH LENGTH 故意压到 128K(.ld:28)，固件超界即触发 'region FLASH overflowed' 硬错误，防止静默覆盖 0x08020000 起的文件存储区。设计意图明确且正确

测试缓冲安全

test_i2c_scan.c:65 'pos+5>=64' 守卫防 buf[64] 溢出；test_pmic.c buf[15] ERR 路径最大写到 index13(14B) 在界内。SCAN_ADDR_BUF_SIZE=127 容纳 Scan 最多 126 个地址(addr 1..126)，安全

数据流 / 控制流

流程
构建流: C_SOURCES(Core+BSP+App+Test+Flash/AW+SPL) → 每个 .o 经 vpath 编译 → 链接 .elf(gc-sections 裁剪未引用段) → objcopy 出 .bin → size 打印。flash 目标用 OpenOCD program .elf verify reset。测试流(片上): test_config.h 宏=1 → main.c #if 接入 Poll → BSP/App 公开 API → UART 上报。测试流(PC): pyserial 打开串口 → build_frame(CRC16) → ser.write → recv_control_frame 解析(自动跳过 0xBB 流帧) → 断言。

模块耦合关系

Makefile 依赖 SPL_DIR 只读库与 Core/BSP/App/Test/Flash 全部源；链接脚本与 startup .s(提供 _sidata/_sdata/_estack 等符号)及 app_flashstore.h 的 0x08020000 边界约定耦合。Test/ 仅调用 BSP_I2Cx_Scan/BSP_I2C3_ReadReg/BSP_UART_Transmit/BSP_GetTick 等公开 API，符合分层。PcTest 脚本与 app_protocol.h 的帧常量/命令码/CRC 算法强耦合(各脚本独立复制实现，存在重复)。

本模块审查发现

中 PC 测试脚本默认波特率 115200 与固件默认 460800 不匹配
📍 PcTest/firmware_test_protocol_0331.py:36,726; firmware_test_i2C_command_0329.py:29; vs bsp_uart.h:17
问题：固件 BSP_UART_BAUDRATE_DEFAULT=460800U(CLAUDE.md §2 亦标注 460800 默认)，但所有 PC 主测脚本 CFG['baud']/argparse default 均为 115200。开箱即用直接运行脚本会因波特率不符而完全收不到固件响应(心跳超时),需用户每次手动 --baud 460800。这是常见的联调踩坑点。
建议：将脚本默认波特率改为 460800 与固件出厂值对齐；或在脚本启动时先试 460800、失败再回退低速并打印明确提示。
中 outline.py 未指定 UTF-8 编码，对含中文的 PcTest 脚本崩溃
📍 outline.py:3-4 (open(sys.argv[1]) 无 encoding)
问题：CLAUDE.md §12 强制要求读 Python 文件前先跑 python outline.py <file>，但 outline.py 用 open() 默认 GBK 编码(Windows)，遇到 PcTest 脚本里的中文 docstring 抛 UnicodeDecodeError('gbk' codec can't decode byte 0xae)，导致该工作流对本项目所有含中文的 .py 完全不可用。
建议：改为 open(sys.argv[1], encoding='utf-8')。
低 PC 脚本 CRC 函数名 'crc16_modbus' 误导(实际非标准 MODBUS)
📍 PcTest/firmware_test_protocol_0331.py:54-55; firmware_test_i2C_command_0329.py:46-47; test_pmic_cmd09_0420.py:15
问题：函数命名/docstring 称 'CRC16-MODBUS poly=0x8005 LSB-first'，但标准 MODBUS 的 LSB-first(右移)实现必须用反射多项式 0xA001;用 0x8005 右移是项目自定义算法。固件 app_protocol.h:9-10 已明确标注 'NOT standard CRC16-MODBUS'。算法本身与固件一致(实测匹配)，仅命名误导，新人易据名错改成 0xA001 而导致整链路 CRC 失配。aw_passthru 脚本(:56)命名已正确标注'与 firmware 一致'。
建议：将三个脚本里的 crc16_modbus 重命名为 crc16_proto 并修正 docstring 为 '项目自定义 CRC16, poly=0x8005 右移, 非标准 MODBUS'，与 firmware/aw 脚本口径统一。
低 四个 PC 脚本各自重复实现 CRC16/build_frame/parse_frame
📍 PcTest/firmware_test_protocol_0331.py:54-68 等四处
问题：crc16、build_frame、parse_frame 在 4 个脚本中各复制一份。协议帧格式或 CRC 一旦演进(协议已到 v2.x，aw 脚本标注 v2.5)，需同步改 4 处，极易漏改造成脚本间行为不一致。属坏味道,非功能 bug。
建议：抽出 PcTest/proto_common.py(crc16/build_frame/parse_frame/recv_control_frame)供各脚本 import，单点维护。
风格 CLAUDE.md §3 文档与 Makefile 实际宏/启动文件名不一致
📍 Makefile:104 (-DSTM32F429_439xx), :83 (startup_stm32f429_439xx.s) vs CLAUDE.md §3 'STM32F429xx' / §4 'startup_stm32f429zgtx.s'
问题：CLAUDE.md 列预定义宏为 STM32F429xx、启动文件为 startup_stm32f429zgtx.s(HAL/CubeMX 命名)，而 Makefile 用 SPL V1.9.0 正确命名 STM32F429_439xx 与 startup_stm32f429_439xx.s。已核 SPL 设备头 stm32f4xx.h:61/70 只识别 STM32F429_439xx，故 Makefile 正确、文档不精确。实测编译通过，非缺陷,仅为防止后续依据文档误改。
建议：更新 CLAUDE.md §3/§4 的宏名与启动文件名为 SPL 实际值,或加注'SPL 命名'说明。
