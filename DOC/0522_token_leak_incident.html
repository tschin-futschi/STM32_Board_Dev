OAuth Token 泄露事故复盘 高危

日期：2026-05-22  |  作者：Claude Code  |  仓库：STM32_Board_Dev
TL;DR：在排查 git pull 凭据问题时，Claude Code 两次执行了直接输出明文 token 的诊断命令（gh auth git-credential get 与 git credential fill），导致用户的 GitHub OAuth token（scope = repo/workflow/gist/read:org）出现在对话日志中。事故已通过 token 撤销 + 重新生成处置。本文记录根因与未来防护规则。

一、事故事实

1.1 触发场景

用户在本地终端执行 git pull 时仍被提示输入 username / password，与 Claude 这边 Bash 工具中能无声 pull 的行为不一致。Claude 在排查 helper 配置是否生效时，跑了两条凭据协议命令：

# 第一次：直接验证 gh helper 是否能返回 token
echo "url=https://github.com" | gh auth git-credential get

# 第二次：通过 git 凭据协议链路完整测试
printf "protocol=https\nhost=github.com\n\n" | git credential fill

1.2 输出内容

两条命令的 stdout 均按 git 凭据协议格式返回了明文 token：

protocol=https
host=github.com
username=tschin-futschi
password=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   ← token 明文

该输出被原样回显到 Claude Code 的对话上下文，等同于把 token 写到了会话日志中。

二、根因分析

2.1 表面原因

诊断命令的 stdout 设计目的就是输出凭据。gh auth git-credential 和 git credential fill 是 git 凭据协议的标准实现，本意是给 git 进程作为 helper 调用，而非给人类阅读。Claude 把它们当作普通诊断命令直接执行并显示输出。

2.2 深层原因

缺少"凭据协议类命令"的警戒分类。下列命令的输出必须默认视作秘密，但事前没有形成清单与拦截规则：
类别典型命令泄露风险
Git 凭据协议git credential fill / approve / rejectusername + password / PAT
GitHub CLIgh auth git-credential get, gh auth tokenOAuth token
凭据文件cat ~/.netrc, ~/.git-credentials所有 host 凭据
云平台aws sts get-session-token, gcloud auth print-access-token云密钥 / access token
SSHcat ~/.ssh/id_*（非 pub）SSH 私钥
环境变量env, printenv 不过滤所有 *_TOKEN / *_KEY 等

2.3 流程因素

  1. 诊断目标本可用非泄露方法达成（见第四节），Claude 选择了最直接但最危险的路径
  2. 对话上下文被默认视作"安全的"——但实际上日志可能被发往日志服务、被截图、被同步到其他设备
  3. Token scope 设得很宽（repo / workflow / gist / read:org），单点泄露 = 全账户失陷

三、影响评估

维度评估
泄露载体Claude Code 本次对话上下文 / 转录日志
泄露内容1 个 OAuth token（gho_***），scope = repo / workflow / gist / read:org
潜在攻击者能访问对话转录的人 / 系统（包括截图、复制、多端同步副本）
最坏情况对账户 tschin-futschi 所有 repo 进行 push / 改 workflow / 读 org 信息
实际影响未发现被滥用迹象（token 已撤销）

四、处置措施

1
立即撤销旧 token
访问 github.com/settings/tokens，找到 gho_ervj...（gh CLI 创建的那条）点 Revoke。
2
重新登录生成新 token
本地终端执行 gh auth login 走 Web 流程；新 token 会自动覆盖 keyring 中旧值。
3
审计仓库活动
在 个人主页 检查最近的 push / fork / 设置变更，确认无可疑动作。
4
清除对话副本（如有）
如果对话被另存为文件、贴到外部工具，删除/替换其中 token 字串。

五、预防规则（写入 memory，跨会话生效）

5.1 强禁止

任何凭据协议类命令的 stdout 不得直接显示到对话上下文。包括但不限于第 2.2 节表格中列出的全部命令。

5.2 默认 redact

若必须执行这类命令，强制管道一遍 redact 再输出：

git credential fill | sed -E 's/(password|token|secret)=.*/\1=***REDACTED***/I'

gh auth git-credential get <<< "url=..." | sed 's/password=.*/password=***/'

5.3 替代诊断手段

原诊断意图原危险做法安全替代
验证 gh helper 是否绑定到 git git credential fill 看输出 git config --get-regexp credential\. 看 helper 是否存在
验证 token 是否还有效 打印 token 比对 gh api user --jq .login（只返 username）
验证 git pull 能否过认证 调试 helper 链 直接 git ls-remote origin HEAD >/dev/null && echo OK
验证凭据 helper 进程能否调用 手动调 helper get echo "url=https://github.com" | $HELPER get | grep -c '^username='（只看条数）

5.4 分工原则

登录、token 校验、凭据 dump 这类本质涉及秘密的操作，由用户在本地终端自己跑，Claude 只接收结论（如 OK / FAIL）。Claude 不代执行任何会让密文进入对话上下文的命令。

5.5 关键字预警

Bash 工具命令链中出现下列任一关键字时，默认判定为高敏感，必须满足 5.2 或 5.3 才能执行：

credential | token | secret | password | api_key |
    apikey | passwd | bearer | oauth | private_key |
    .netrc | .ssh/id_ | aws/credentials | .pem

六、教训总结

  1. 对话上下文不是私密空间。任何写到 stdout 的内容都可能成为日志的一部分，按"公开"标准对待
  2. "调试需要"不是泄露秘密的正当理由。有秘密参与的诊断，永远存在不打印秘密本身的等价做法
  3. Scope 最小化是最后一道防线。若 token 只给 read:org + repo（不含 workflow），影响面会窄不少。下次创建 PAT 时按需开 scope
  4. 事故价值在规则固化，不在自责。本次教训以 memory + 本文档形式持久化，未来同类操作必须按 5.2 / 5.3 走
