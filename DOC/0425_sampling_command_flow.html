1. 问题本质：单线程 + 单 UART 的并发挑战

系统只有一个主循环线程和一条 UART 通道，但需要同时完成两件事：

任务方向帧类型频率
数据采样上报STM32 → PC 0xBB 数据流帧 500~5000 Hz
控制指令交互PC ↔ STM32 0xAA55 控制帧 不定时（心跳 1Hz + 用户操作）

核心矛盾：两种帧共享同一个 DMA TX 缓冲区，同一时刻只能发一帧。如何保证：

  • 控制指令的响应不被采样帧"饿死"
  • 采样帧不会覆盖正在发送的控制帧
  • I2C 总线操作不会冲突

2. 主循环调度模型

主循环非常简洁，每次迭代做四件事：

/* main.c — 主循环 */
while (1)
{
    uint32_t now = BSP_GetTick();

    App_Protocol_Poll();      // ① 解析 RX 缓冲区中的控制帧，执行命令
    App_Sample_Poll();        // ② 如果 TIM6 flag 置位，执行采样+发送

    // ③ PMIC HWEN 事件处理
    if (g_pmicHwenFlag != 0U) { ... }

    // ④ LED 心跳翻转（200ms）
    if (now - prevTick >= HEARTBEAT_INTERVAL_MS) { ... }
}
关键观察：App_Protocol_Poll() 在 App_Sample_Poll() 之前执行。 这意味着每次主循环迭代中，控制指令总是先于采样帧被处理。控制帧天然具有"软优先级"。

App_Protocol_Poll() 的工作

void App_Protocol_Poll(void)
{
    // 1. 检查 I2C 总线恢复事件
    if (BSP_I2C2_GetAndClearRecoveryFlag() != 0U)
        App_Protocol_SendDebugInfo("I2C bus recovered");

    // 2. 检查 RX 缓冲区溢出
    overflow = BSP_UART_GetAndClearOverflowCount();
    if (overflow > 0U)
        App_Protocol_SendDebugInfo(...);

    // 3. 批量取出 RX 缓冲区中的所有字节
    len = BSP_UART_RxRead(s_rxBuf, BSP_UART_RX_BUF_SIZE);

    // 4. 逐字节送入状态机
    for (i = 0U; i < len; i++)
        ProcessByte(s_rxBuf[i]);  // → 帧完整时调用 DispatchFrame()
}

App_Sample_Poll() 的工作

void App_Sample_Poll(void)
{
    // 0. 两个消费者都空闲 → 直接返回
    if (s_sampling == 0U && s_genMode == GEN_NONE) return;

    // 1. 检查 TIM6 中断标志
    if (BSP_SampleTim_GetFlag() == 0U) return;
    BSP_SampleTim_ClearFlag();

    // 2. Phase 1: 波形发生器写入（如果在运行）
    if (s_genMode != GEN_NONE) { ... Generator_DoWrite(); }

    // 3. Phase 2: 采样读取 + 发送
    if (s_sampling != 0U)
    {
        if (BSP_UART_IsTxBusy() != 0U) return;  // ★ TX 忙则跳过本帧
        SendStreamFrame(effectiveMask);
    }
}

3. TX 通道仲裁：谁让谁？

系统中有两类发送者，它们对"TX 正忙"的处理策略截然相反：

发送者函数TX 忙时行为策略
控制帧 SendFrame() 阻塞等待 BSP_UART_TxWait() 可靠性优先：必须发出去
数据流帧 SendStreamFrame() 直接跳过 return 实时性优先：丢帧好过阻塞

这一设计的精妙之处在于：

控制帧永远不会被"饿死"：SendFrame() 会等到 DMA 发完再发，所以控制帧的响应一定会送达 PC。

数据流帧主动让路：如果上一帧还在发，数据流帧丢弃当前采样，等下一个 TIM6 tick 再尝试。对于示波器场景，偶尔丢一帧完全可接受。

来看具体的代码对比：

/* 控制帧发送 — app_protocol.c:119-147 */
static ErrorStatus SendFrame(uint8_t seq, uint8_t cmd,
                              const uint8_t *pData, uint8_t len)
{
    // ...构建帧...

    BSP_UART_TxWait();              // ★ 阻塞：等上一帧 DMA 发完
    return BSP_UART_Transmit(s_txBuf, idx);
}

/* 数据流帧发送 — app_sample.c:316-321 */
if (s_sampling != 0U)
{
    if (BSP_UART_IsTxBusy() != 0U) return;  // ★ 非阻塞：TX 忙则放弃
    SendStreamFrame(effectiveMask);
}

4. 完整时序图：采样中收到心跳

最常见的场景：采样运行中，PC 每秒发一次心跳（CMD=0x00）。

时间轴 ──────────────────────────────────────────────────────► TIM6 ISR ↓flag ↓flag ↓flag │ │ │ 主循环迭代 ├──iter N──┤ ├──iter N+1──┤ ├──iter N+2──┤ ├──iter N+3──┤ Protocol_Poll│ 无RX数据 │ │收到心跳帧 │ │ 无RX数据 │ │ 无RX数据 │ │ (快速返回)│ │→ProcessByte│ │ (快速返回) │ │ (快速返回) │ │ │ │→Dispatch │ │ │ │ │ │ │ │→SendFrame │ │ │ │ │ │ │ │ TxWait: │ │ │ │ │ │ │ │ 等0xBB发完 │ │ │ │ │ │ │ │ DMA发心跳 │ │ │ │ │ │ │ │ │ │ │ │ │ Sample_Poll │ flag=1 │ │ flag=0 │ │ flag=1 │ │ flag=0 │ │ 读I2C │ │ (快速返回) │ │ TX忙(心跳) │ │ (快速返回) │ │ DMA发0xBB│ │ │ │ → 跳过本帧 │ │ │

逐步解析

  1. 迭代 N：TIM6 flag 置位，Sample_Poll 读 I2C 寄存器，构建 0xBB 帧，启动 DMA 发送
  2. 迭代 N+1：Protocol_Poll 从 RX 缓冲区取出心跳帧字节，状态机解析完成，调用 HandleHeartbeat()
    • SendFrame() 内部先调用 BSP_UART_TxWait()，等迭代 N 的 0xBB DMA 发完
    • 然后启动新的 DMA 发送心跳响应帧
  3. 迭代 N+2：TIM6 flag 又置位，Sample_Poll 准备发 0xBB，但心跳帧 DMA 还在进行
    • BSP_UART_IsTxBusy() 返回 1 → 跳过本帧
    • 这一采样点的数据丢弃，PC 端示波器少收一个点（可接受）
  4. 迭代 N+3：DMA 空闲，下次 TIM6 flag 到来时正常采样
代价分析：一次心跳交互大约导致丢失 1~2 个采样帧（取决于波特率和采样频率）。 以 115200 bps、1kHz 采样为例，一个 9 字节心跳帧的 DMA 传输约 0.8ms，而采样间隔是 1ms，所以最多丢 1 帧。

5. 完整时序图：采样中收到寄存器读取

比心跳复杂一些：HandleReadReg() 需要先做一次 I2C 读操作，然后再发送响应帧。

时间轴 ──────────────────────────────────────────────────────────► Protocol_Poll ┌─────────── 收到 0x20 ReadReg 命令 ───────────┐ │ ProcessByte × 9 (9字节控制帧) │ │ → DispatchFrame → HandleReadReg │ │ ┌──────────────────────────────────────┐ │ │ │ ① I2C2 读寄存器 (~142μs) │ │ │ │ BSP_I2C2_ReadReg(addr, reg, ...) │ │ │ │ ② 构建响应帧 │ │ │ │ SendFrame(seq, 0x20, resp, 2) │ │ │ │ TxWait: 等 0xBB DMA 发完 │ │ │ │ DMA 启动: 发 11 字节控制响应帧 │ │ │ └──────────────────────────────────────┘ │ └──────────────────────────────────────────────┘ ↓ 返回主循环 Sample_Poll flag可能已置位 → TX忙，跳过 │ 下个tick → 正常发送

这里涉及一个重要的细节：I2C 总线竞争。

HandleReadReg() 通过 App_Motor_ReadReg() 访问 I2C2 总线。 如果此时 Sample_Poll() 也要读 I2C2（通过 SendStreamFrame() 中的 App_Motor_ReadReg()），会不会冲突？

答案：不会。因为整个系统是单线程协作式的：

  1. Protocol_Poll() 先执行，HandleReadReg 的 I2C 读在主循环内完成
  2. 返回后才轮到 Sample_Poll()，此时 I2C 总线已空闲
  3. 两者不可能并行执行，因为没有抢占式调度

6. 特殊场景：采样中请求 BulkRead

HandleBulkRead(0x22) 是最复杂的命令——它需要连续读取数百个寄存器，分多个包发送。 这与高频采样有严重冲突：

问题：BulkRead 可能需要几秒才能完成（读 256 个寄存器 × 142μs ≈ 36ms，加上多包发送）。 这期间 TIM6 中断不断触发，采样帧无法发送；PC 的心跳也可能超时。

代码的解决方案：直接拒绝。

/* app_protocol.c:455-460 — BulkRead 互斥检查 */
static void HandleBulkRead(const Proto_Frame_t *pFrame)
{
    ...
    /* Reject bulk read while sampling is active */
    if (App_Sample_IsActive() != 0U)
    {
        SendErrorResp(pFrame->seq, PROTO_ERR_EXEC_FAIL);
        return;
    }
    ...
}
设计决策：BulkRead 和采样互斥。PC 必须先停止采样（0x51），再发 BulkRead（0x22）， 读完后再启动采样（0x50）。这避免了复杂的时分复用逻辑。

BulkRead 自身的心跳维持

虽然采样被禁止，但 BulkRead 期间 PC 仍在发心跳。如果不响应，PC 会判定连接断开。 代码用了一个精巧的递归 Poll 机制：

/* app_protocol.c:462-498 — BulkRead 多包发送循环 */
s_bulkReadActive = 1U;           // ★ 设置互斥标志
for (pktIdx = 0U; pktIdx < totalPkts; pktIdx++)
{
    // ...读寄存器，构建包...

    BSP_UART_TxWait();             // 等上一包发完
    App_Protocol_Poll();           // ★ 处理 RX：只允许心跳通过
    SendFrame(...);                // 发送本包
}
s_bulkReadActive = 0U;           // 清除互斥标志

/* app_protocol.c:734-740 — 分发器中的 BulkRead 保护 */
static void DispatchFrame(const Proto_Frame_t *pFrame)
{
    /* During bulk read, only respond to heartbeat */
    if (s_bulkReadActive && (pFrame->cmd != PROTO_CMD_HEARTBEAT))
    {
        return;  // 非心跳命令全部忽略
    }
    ...
}
BulkRead 期间的调用栈 App_Protocol_Poll() ← 主循环调用（外层） └→ ProcessByte() × N └→ DispatchFrame() └→ HandleBulkRead() ├→ 读寄存器 + 发包 ├→ App_Protocol_Poll() ← 内层递归调用 │ └→ ProcessByte() × N │ └→ DispatchFrame() │ ├→ 心跳 → HandleHeartbeat() ✓ │ └→ 其他 → 忽略（s_bulkReadActive=1） ├→ 读寄存器 + 发包 └→ ...重复...
栈安全：递归只会发生一层（内层 BulkRead 被 s_bulkReadActive 阻止）， 额外栈消耗约 300~400 字节，2KB 栈够用。

7. 特殊场景：采样中切换波特率

这是一个有副作用的命令：波特率切换会影响正在进行的 DMA 发送和 RX 接收。

/* app_protocol.c:220-245 — HandleSetBaudrate */
static void HandleSetBaudrate(const Proto_Frame_t *pFrame)
{
    ...
    // ① 用旧波特率发送 ACK
    SendFrame(pFrame->seq, PROTO_CMD_SET_BAUDRATE, NULL, 0U);

    // ② 等 DMA 发完 + 移位寄存器排空
    BSP_UART_TxWait();

    // ③ 切换波特率（内部会禁用 USART、重新初始化）
    BSP_UART_SetBaudrate(k_baudrateTable[idx]);

    // ④ 丢弃过渡期间的垃圾字节
    BSP_UART_RxFlush();

    // ⑤ 重置状态机，防止半帧残留
    s_state = STATE_WAIT_SOF1;
}

采样中执行波特率切换的时序：

  1. SendFrame() 内部 TxWait() 等当前 0xBB 帧 DMA 完成
  2. 用旧波特率 DMA 发出 ACK 控制帧
  3. 再次 TxWait() 确保 ACK 的最后一个 bit 离开移位寄存器
  4. BSP_UART_SetBaudrate() 内部：禁用 RXNE 中断 → 禁用 DMA → 关闭 USART → 以新波特率重新初始化
  5. 切换完成后，采样帧继续以新波特率发送

采样不会中断：TIM6 继续运行，s_sampling 未被修改。 只是在波特率切换那一瞬间（~1ms），有 1~2 个采样帧被跳过。

8. 特殊场景：采样中启动波形发生器

波形发生器（Linear/Cosine）和采样共享 TIM6，可以同时运行。 但它们的 I2C 操作需要在同一个 tick 内完成，否则会丢帧。

Tick 超限检查（80% 规则）

/* app_protocol.c:514-531 — CheckTickOverrun */
static uint8_t CheckTickOverrun(uint8_t readCh, uint8_t writeCh)
{
    uint32_t periodUs   = BSP_SampleTim_GetPeriodUs();
    uint32_t estimateUs = readCh  * I2C_READ_US      // 142μs/次
                        + writeCh * I2C_WRITE_US;     // 115μs/次
    uint32_t limitUs    = (periodUs * 80U) / 100U; // 80% 裕量

    if (estimateUs > limitUs)
    {
        // 发送 Debug 信息告知 PC 工作量超限
        App_Protocol_SendDebugInfo("Tick overrun: ...");
        return 0U;  // 拒绝
    }
    return 1U;  // 允许
}

举个例子：1kHz 采样（周期 1000μs），80% 裕量 = 800μs：

场景读通道写通道预计耗时结果
4通道采样 + 无发生器40 4 × 142 = 568μs允许
4通道采样 + 1路线性发生器41 4 × 142 + 1 × 115 = 683μs允许
4通道采样 + 3路余弦发生器43 4 × 142 + 3 × 115 = 913μs拒绝（超 800μs）

两阶段 Poll 的执行顺序

一个 TIM6 tick 内的工作分配 ┌────────────────────────────────────────────────┐ │ App_Sample_Poll() │ │ │ │ Phase 1: 波形发生器写入 │ │ ├─ 检查 divider 计数器 │ │ └─ 调用 App_Motor_WriteReg() × writeCh │ │ (I2C2 写寄存器, ~115μs/次) │ │ │ │ Phase 2: 采样读取 + 发送 │ │ ├─ 检查 TX busy → 跳过 or 继续 │ │ ├─ 调用 App_Motor_ReadReg() × readCh │ │ │ (I2C2 读寄存器, ~142μs/次) │ │ └─ 构建 0xBB 帧 → BSP_UART_Transmit() │ │ │ │ 总耗时 = writeCh×115 + readCh×142 (μs) │ └────────────────────────────────────────────────┘
先写后读的意义：发生器先写入新值到电机 IC 寄存器，采样紧接着读取同一组寄存器。 这样 PC 端示波器能立即看到写入的效果，没有一个 tick 的延迟。

9. I2C 总线竞争分析

项目中有 3 条 I2C 总线，电机 IC 相关操作都走 I2C2：

操作I2C 总线触发源
采样读寄存器I2C2TIM6 tick → Sample_Poll
发生器写寄存器I2C2TIM6 tick → Sample_Poll
HandleReadReg (0x20)I2C2PC 命令 → Protocol_Poll
HandleWriteReg (0x21)I2C2PC 命令 → Protocol_Poll
HandleMotorPing (0x05)I2C2PC 命令 → Protocol_Poll
PMIC 操作 (0x08-0x0A)I2C3PC 命令 → Protocol_Poll
I2C Scan (0x07)I2C1/2/3PC 命令 → Protocol_Poll

为什么不会冲突？

关键在于执行模型是协作式的（cooperative），不是抢占式的：

/* 主循环 — 顺序执行，无并行 */
while (1)
{
    App_Protocol_Poll();   // 如果有命令，在这里完整执行（含 I2C 操作）
                           // 执行完毕后才返回

    App_Sample_Poll();    // 此时 Protocol_Poll 已完成，I2C 总线空闲
                           // 可以安全地进行采样 I2C 操作
}

同一条 I2C 总线上的操作永远不会交错，因为：

  • I2C 操作是同步阻塞的（轮询模式 + 超时保护）
  • 主循环是顺序执行的，Poll A 执行完才轮到 Poll B
  • ISR 中不做 I2C 操作（只置标志），不会从中断上下文访问总线
