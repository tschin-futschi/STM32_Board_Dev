# 开发总结 2026-03-26 23:00

## 变更清单

### 修改文件

- **`App/Src/app_protocol.c`** — Code Review 修复
  - 新增编译期静态断言 `_Static_assert(PROTO_MAX_FRAME_LEN <= BSP_UART_TX_BUF_SIZE)`，防止协议帧长超过 DMA 发送缓冲区时静默截断
  - CRC16-MODBUS 初始值 `0xFFFFU` 两处裸写改为命名常量 `k_crc16Init`，消除魔法数字
  - `SendFrame()` 中描述"做了什么"的 3 行无效注释删除，保留字节序说明注释
  - `s_rxBuf[BSP_UART_RX_BUF_SIZE]` 从 `App_Protocol_Poll()` 函数内部提升至文件作用域，与其他 `s_` 静态变量保持一致

- **`BSP/Inc/bsp_uart.h`**
  - `BSP_UART_TX_BUF_SIZE` 从 `256U` 改为 `264U`（>= `PROTO_MAX_FRAME_LEN` 262），使静态断言通过，同时覆盖协议最大帧长

## 修复依据（Review 发现的问题）

| 优先级 | 问题 | 处理 |
|--------|------|------|
| P0 | `PROTO_MAX_FRAME_LEN`（262）> `BSP_UART_TX_BUF_SIZE`（256），大数据帧会被 BSP 静默拒绝 | 已修复：扩容 + 静态断言 |
| P1 | `0xFFFFU` CRC 初始值两处裸写，违反魔法数字规则 | 已修复：`k_crc16Init` |
| P2 | `SendFrame` 中多余注释解释"做了什么" | 已修复：删除 |
| P2 | `s_rxBuf` 隐藏在函数体内，与文件其他静态变量风格不一致 | 已修复：提升至文件作用域 |
| 误报 | `STATE_WAIT_DATA` 越界风险——`PROTO_MAX_DATA_LEN=255` 等于 uint8_t 上限，实际安全 | 跳过 |
