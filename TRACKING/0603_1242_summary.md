# 0603_1242 构建系统审查与改进总结

## 背景

开发者要求审查构建系统(Makefile / 链接脚本 / 启动文件配置),并据此做了 4 项改动。
全程以 **clean build + bin 哈希逐字节比对** 为验收手段,确保不影响实际烧录固件。

---

## 变更清单

### 1. `Linker/STM32F429ZGTX_FLASH.ld` —— 删除无效 `/DISCARD/` 块

删除文件末尾的:
```ld
/DISCARD/ :
{
    libc.a(*)
    libm.a(*)
    libgcc.a(*)
}
```
**原因(经实证)**:该块意图丢弃 libc/libm/libgcc,实际**什么都没丢**:
- 前面 `.text : { *(.text*) }` 等通配已贪婪抢占这些库的 section,轮到 `/DISCARD/` 时已无可丢;
- nano.specs 下真正链接的是 `libc_nano.a`,根本不匹配 `libc.a` 模式。

**验证**:删除前(HEAD,含该块)与删除后,`firmware.bin` 哈希**逐字节相同**
(`b67566735fe94ab77075082a2d9fc6887bada1603e3ce4dd9f3440604d80924e`),证明确为死代码。

### 2. `Makefile` —— 加头文件依赖追踪 `-MMD -MP`

- `CFLAGS += -MMD -MP`(编译时额外产出 `.d` 依赖文件)
- 文件末尾 `.PHONY` 前加 `-include $(OBJECTS:.o=.d)`

**原因**:原 Makefile 无依赖追踪,改任何 `.h` 都不触发依赖它的 `.c` 重编译,只能 `make clean` 全量重建,易得陈旧固件。
**影响**:只额外生成 `.d` 文件(本次 33 个),不改变 `.o`/`.elf`/`.bin` 字节。

### 3. `Makefile` —— 新增 Intel HEX 产物

- `all` 目标追加 `$(BUILD_DIR)/$(TARGET).hex`
- 新增规则:
```makefile
# Intel HEX
$(BUILD_DIR)/$(TARGET).hex: $(BUILD_DIR)/$(TARGET).elf
	$(CP) -O ihex $< $@
```
**原因**:开发者需要 hex 格式,供 ST-Link Utility / J-Flash / STM32CubeProgrammer 等只认 hex 的烧录器使用。
**验证**:`firmware.hex` 正常生成(95392 字节),首行 `:020000040800F2`(基地址 `0x0800_0000` 正确),末行 `:00000001FF`(标准 EOF);加入后 `.bin` 哈希仍为 `b67566…`,对原产物零影响。

### 4. `CLAUDE.md` —— 修正两处文档与代码不一致

| 位置 | 原文 | 改为 | 说明 |
|------|------|------|------|
| §3 预定义宏 | `STM32F429xx` | `STM32F429_439xx` | Makefile 实际用 `-DSTM32F429_439xx`(SPL 设备族宏);SPL 头文件只认此宏,Makefile 正确,文档错 |
| §4 目录树 | `startup_stm32f429zgtx.s` | `startup_stm32f429_439xx.s` | 实际文件名 |
